## Context

Thai government raised SSO salary ceiling from 15,000 ฿ to 17,500 ฿ effective 1 Jan 2026 — employee SSO contribution caps at 875 ฿/month (was 750 ฿). The codebase has 750/15000 hard-coded in DTO `@Max`, payroll calculator, tests, and CoA CSV comments. Lift these to a config table with `effective_from`/`effective_to` so future ceiling changes (2572 → 1000, 2575 → 1150) don't require code edits.

**Legal urgency:** any payroll posted for May 2026 close that uses 750 cap is non-compliant with the new กฎกระทรวง.

## Open Questions

- [x] Q: Use new `sso_config` table or extend existing `system_config` key-value store? — **Dedicated `SsoConfig` table**. The k-v store doesn't model multi-row stepped periods naturally.
- [x] Q: `payment_date` lookup — pin to the payment date in DTO, or to `documentDate` of the EX_PAYROLL row? — **`dto.documentDate`**. That's the legal posting date; `payrollPeriod` is a label that may shift while `documentDate` is committed.
- [x] Q: Backward compat for in-flight payroll docs created with 750 cap before this PR lands — leave as-is or re-stamp? — **Leave as-is**. Validation is enforced only on new writes via `SsoConfigService.validateContribution`. Historical rows retain whatever value they were posted with — re-stamping would require running an adjusting JE on every affected payroll doc, which isn't worth it for a value that was lawful at the time.
