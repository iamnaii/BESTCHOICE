#### Warning — Financial `Number()` casts in override JV form (frontend)

**File:** `apps/web/src/pages/other-income/components/EditableJournalTable.tsx`

```tsx
// Line ~3914-3915: snapshot creation
debit: Number(Number(l.debit).toFixed(2)),
credit: Number(Number(l.credit).toFixed(2)),

// Line ~4293, 4303: form input onChange
onChange={(e) => updateLine(idx, { debit: Number(e.target.value) || 0, credit: 0 })}
onChange={(e) => updateLine(idx, { credit: Number(e.target.value) || 0, debit: 0 })}
```

**Risk:** Low — JS `Number` is fine for 2-decimal financial values up to ~10^15. The backend safely wraps incoming values with `new Decimal(l.debit)` before any arithmetic. No precision loss is expected for typical THB amounts. However, this pattern is inconsistent with the Decimal convention and could silently propagate floating-point artifacts if amounts are large or multi-step.

**Suggested fix:** Use string values for `debit`/`credit` in `EditableJournalLine` interface and pass `e.target.value` directly; let the backend Decimal constructor parse the string.

#### Info — `Number(it.whtPct)` in validation service

**File:** `apps/api/src/modules/other-income/services/validation.service.ts` (line ~99)

```ts
const pct = Number(it.whtPct);
if (!VALID_WHT_PCT.includes(pct)) { ... }
```

Used only for membership check against `[0,1,2,3,5,7,10,15]` — no arithmetic. Safe as-is, but `it.whtPct.toNumber()` (Prisma Decimal method) would be more idiomatic.

#### Info — Large frontend page (>500 lines)

`OtherIncomeEntryPage.tsx` and `EditableJournalTable.tsx` are both large components introduced in this branch. Not a blocker, but future refactors may want to extract sub-components.
