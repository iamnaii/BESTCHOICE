#### 🔴 Critical

**C1 — `Number()` arithmetic on Decimal financial fields (financial precision risk)**

Multiple new service methods perform arithmetic on `Prisma.Decimal` money fields after converting them with `Number()`, which loses precision for large values (>2^53) and produces float rounding errors.

| File | Line | Issue |
|------|------|-------|
| `apps/api/src/modules/accounting/accounting.service.ts` | ~2097 | `const remaining = Number(p.amountDue) - Number(p.amountPaid ?? 0)` — used in bad-debt aging bucket summation |
| `apps/api/src/modules/notifications/scheduler.service.ts` | ~119, 241 | `sum + (Number(p.amountDue) - Number(p.amountPaid) + Number(p.lateFee))` — overdue total for LINE notification |
| `apps/api/src/modules/notifications/scheduler.service.ts` | ~128 | `contract.payments.reduce((sum, p) => sum + Number(p.lateFee), 0)` |
| `apps/api/src/modules/contracts/contracts.service.ts` | ~1221 | `(acc, c) => acc + Number(c.financedAmount)` — portfolio sum |
| `apps/api/src/modules/accounting/bank-reconciliation.service.ts` | ~126, 140 | `Number(p.amountPaid)` passed to `amountMatches()` — tolerance comparison |

**Required fix:** Replace with `Prisma.Decimal` arithmetic. For bucket summation use `new Prisma.Decimal(0)` accumulator + `.add()`. For `amountMatches()` the signature should accept `Prisma.Decimal` instead of `number`.

**Exception note:** `documents.service.ts` and `contracts.service.ts` structured-logger calls also convert Decimal to Number for display/HTML generation only — lower severity but still violates the rule; use `.toString()` or `.toFixed(2)` instead.
