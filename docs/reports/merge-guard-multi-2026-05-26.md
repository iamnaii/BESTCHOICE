### What Changed

- **Before:** `ContractExchangeService.approve()` ran the full JE chain (A.1–A.4), flipped old contract + old product status, and returned JE IDs — all atomically.

### Warning Issues

**W1 — `as any` on Prisma transaction client** _(contract-exchange.service.ts, lines ~739 and ~820)_

```ts
// line ~739
const request = await (tx as any).contractExchangeRequest.findFirst({ ... });

// line ~820
await (tx as any).contractExchangeRequest.update({ ... });
```

`Prisma.TransactionClient` should expose `contractExchangeRequest` — the `as any` cast is a workaround for a type resolution issue (likely a missing re-export in the module). If Prisma is regenerated or the schema changes, this silently breaks with a runtime error (`TypeError: tx.contractExchangeRequest is not a function`) rather than a compile-time error. The path is transaction-critical (JE chain finalization). **Should replace with a properly typed transaction parameter or a local type extension.**

**W2 — `(contract as any).exchangedFromContractId`** _(contract-workflow.service.ts, lines ~1054 and ~1095)_

```ts
const isExchangeContract = !!(contract as any).exchangedFromContractId;
// ...
exchangedFromContractId: (contract as any).exchangedFromContractId,
```

The `contract` object returned by `prisma.contract.findUnique` may or may not include `exchangedFromContractId` depending on the select/include. Using `as any` silently returns `undefined` if the field is not selected, causing `isExchangeContract = false` and silently skipping `finalizeAfterActivation`. **Should add `exchangedFromContractId: true` to the `select` clause in the `findUnique` call and use a typed interface.**

**W3 — `data: { ... } as any` on Prisma update calls** _(contract-exchange.service.ts, lines ~807–811)_

```ts
data: { status: 'EXCHANGED', exchangedAt: new Date() } as any,
data: { status: 'REFURBISHED', ownedByCompanyId: shopCompanyId } as any,
```

Bypasses Prisma's strict enum + field type checking on update data. If the schema changes (e.g. `ContractStatus.EXCHANGED` removed, or `exchangedAt` field dropped), no compile-time error surfaces. Pattern is pre-existing in older code but should not be extended. **Should remove `as any` and let Prisma validate the types.**

### What Changed

Fixes a PDPA consent constraint violation introduced by `feat/sp2-exchange-sign-flow`:

- `Contract.pdpaConsentId` is `@unique` in the Prisma schema — reusing the old contract's consent row for the new exchange contract would violate this constraint at runtime.
- Fix: `approve()` now clones the consent row via `tx.pDPAConsent.create(...)` (same customer + consent data, fresh UUID) and sets `pdpaConsentId = cloned.id` on the new contract.
- If the old contract has no consent, `pdpaConsentId = null` (no clone attempted).

### Critical Issues

_None_

### Warning Issues

_None_

### Info

**I1 — Clone semantics for PDPA audit trail**

Cloning creates a new consent row with the same `grantedAt`, `purposes`, `consentVersion`, `signatureImage`, etc. This is correct for the unique-constraint fix, but means a single customer consent event now has two rows. Future PDPA audit tooling should join on `customerId + grantedAt` rather than treating each row as an independent consent event. Low urgency — document in PDPA audit if/when it is built.

- 3 new test cases: clone path, no-consent (null) path, and pdpaConsentId differs from original.
- `pDPAConsent.findUnique` is NOT called when `old.pdpaConsentId` is null — correct guard.

Small, focused fix. Correct solution to the unique constraint problem. Well-tested.
