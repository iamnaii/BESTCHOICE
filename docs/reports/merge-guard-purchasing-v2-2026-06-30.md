### Issues

#### 🔴 Critical 1 — `Number()` on Prisma Decimal for money storage
**File**: `apps/api/src/modules/purchase-orders/services/po-receiving.service.ts`  
**Line**: ~416 in diff (`costPrice: Number(poItem.unitPrice)`)

```ts
const product = await tx.product.create({
  data: {
    ...
    costPrice: Number(poItem.unitPrice),  // ← VIOLATION
```

`poItem.unitPrice` is a `Prisma.Decimal` retrieved from the database. Calling `Number()` on it converts to a 64-bit float before writing to `Product.costPrice` (a `@db.Decimal(12, 2)` field). For standard phone prices this is unlikely to corrupt data, but it violates the project rule "use Prisma.Decimal, never Float" and will fail the automated `Number(_sum` check that enforces this convention.

**Fix**:
```ts
costPrice: new Prisma.Decimal(poItem.unitPrice),
```

#### 🔴 Critical 2 — Floating-point accumulation for PO financial totals
**File**: `apps/api/src/modules/purchase-orders/services/po-receiving.service.ts`  
**Line**: ~550 in diff

```ts
const totalAmount = dto.items.reduce(
  (s, i) => s + Number(i.unitPrice) * i.quantity,  // ← float arithmetic
  0,
);
const po = await tx.purchaseOrder.create({
  data: {
    totalAmount,   // stored as Decimal(12,2)
    netAmount: totalAmount,
    ...
  },
});
```

`dto.items` are typed as DTO numbers (JSON-parsed). The `.reduce()` accumulates via JavaScript float arithmetic. For amounts like `12345.67 × 3`, float math can yield `37037.009999999995` instead of `37037.01`, which Postgres rounds to `37037.01` at the Decimal boundary — silently wrong until an audit finds the mismatch.

**Fix** (use Prisma.Decimal throughout):
```ts
const totalAmount = dto.items.reduce(
  (sum, i) => sum.plus(new Prisma.Decimal(i.unitPrice).times(i.quantity)),
  new Prisma.Decimal(0),
);
// totalAmount is now a Prisma.Decimal — pass directly to Prisma
```
