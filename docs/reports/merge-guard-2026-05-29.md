### Analysis

Migrates `FacebookWebhookController` from reading `FB_VERIFY_TOKEN` / `FB_APP_SECRET` directly from environment variables to reading them from `IntegrationConfig` (DB → env fallback via `IntegrationConfigService`). This enables runtime token rotation via the Settings → Integrations UI without requiring a redeploy.

Key changes:
- `verifyWebhook` is now `async`; reads `verifyToken` from `IntegrationConfig`.
- `verifySignature` is now `async`; reads `appSecret` from `IntegrationConfig`.
- `handleDataDeletion` and `handleDeactivation` use new `getAppSecret()` helper.
- `IntegrationsModule` added to `ChatAdaptersModule` imports (required for DI).
- 3 new test cases for `verifyWebhook` (correct token, wrong token, empty token).

### Issues

**Critical:** None

**Warning:**

- **`token === verifyToken` is non-timing-safe** in `verifyWebhook` (line 82 of controller):
  ```ts
  if (mode === 'subscribe' && verifyToken && token === verifyToken) {
  ```
  This is a simple string equality check rather than `timingSafeEqual`. However, this is **pre-existing behavior** (the original code used identical `===` comparison). The security-critical HMAC signature check in `verifySignature` correctly uses `timingSafeEqual` via `timingSafeCompare`. The verify endpoint is a one-time setup call, so the risk is low. No regression introduced.

**Info:**

- `IntegrationConfigService` uses an in-memory TTL cache. After an admin updates the verify token or app secret via the UI, the cached value may remain stale until the TTL expires. If a webhook verification is attempted immediately after a key rotation, the old cached key may still be used. This is a UX/operational concern, not a security regression (env fallback is preserved).
- `verifyWebhook` now correctly **fails closed** when no token is configured (`verifyToken &&` guard added) — improvement over the original code that would have matched against `undefined === undefined` if `FB_VERIFY_TOKEN` was unset.
- Controller is intentionally public (no `JwtAuthGuard`) — matches the documented exception list in `security.md` (webhook receiver, verified by Facebook HMAC on POST).
- No money fields, no DTO changes, no Prisma queries in the diff.

### Recommendation: ✅ APPROVE

Good security improvement. Moves secrets to the DB-backed `IntegrationConfig` with proper env fallback. The `timingSafeEqual` is preserved for HMAC verification. The non-timing-safe token comparison on the verify endpoint is pre-existing and low-risk. New tests provide coverage for the three key behaviors.

**Optional follow-up (non-blocking):** Consider using `timingSafeEqual` for the `verifyToken` comparison to match the consistency of the HMAC check.
