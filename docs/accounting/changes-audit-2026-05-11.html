2.1 ค่าเสื่อมราคา cron ไม่เคยรันจริง (CPA C-1) P0

บัค: TAS 16 ¶49 compliance — งบขาดค่าเสื่อมราคาทุกเดือน

PR #808 · ไฟล์: apps/api/src/modules/journal/cron/depreciation.cron.ts

Cron schedule ใช้ '0 1 28-31 * *' + timeZone: 'Asia/Bangkok' ซึ่ง fire ที่ 01:00 BKK = 18:00 UTC ของวันก่อนหน้า แต่ guard ภายในใช้ new Date().getMonth() ที่อ่าน UTC date — เงื่อนไข "พรุ่งนี้เดือนใหม่หรือยัง" ไม่เคยเป็นจริง → cron return early ทุกครั้ง → ไม่มี JE ค่าเสื่อมโพสต์อัตโนมัติเลย

ผลกระทบ: งบกำไรขาดทุนทุกเดือนขาดค่าเสื่อม, NBV บนงบดุลไม่ลดลง, asset_register ไม่ update accumulatedDepr

แก้: compute "today BKK" + "tomorrow BKK" ด้วย Intl.toLocaleString('en-CA', { timeZone: 'Asia/Bangkok' }) แล้วเทียบเดือน — guard ทำงานถูกต้องที่วันสุดท้ายของเดือน BKK

ฝ่ายบัญชีต้องทำ: ตรวจสอบ JE_DEPRECIATION ของงวดก่อนหน้านี้ (มี.ค.-เม.ย. 2569) ว่ามีโพสต์ครบหรือไม่ ถ้าไม่ครบต้อง post manually ผ่าน endpoint POST /depreciation/run หรือสร้าง JE adjusting

เดิมระบบยอมให้ตั้งหนี้ ACCRUAL ที่มี WHT พร้อมกันใน JE เดียว — ถ้าตั้งหนี้ 30/04 แต่จ่ายจริง 05/05 → WHT จะถูกบันทึกในงวด 04 ผิดวันที่ → ภ.ง.ด.3/53 นำส่งผิดงวด → เบี้ยปรับ

Template เดิมอ่าน asset + สร้าง JE lines นอก $transaction ก่อนเปิด tx — ถ้า depreciation cron วิ่งระหว่างช่วงนี้, accumulatedDepr stale → NBV ผิด → กำไร/ขาดทุนผิด → P&L งวดนั้นผิด

2.5 Depreciation last-month rounding drift (Asset C3)

บัค: 107,000 / 36 = 2,972.22 → 36 × 2,972.22 = 106,999.92 → NBV ค้าง 0.08 ฿

PR #808 · ไฟล์: apps/api/src/modules/journal/cpa-templates/depreciation.template.ts

Straight-line depreciation ใช้ ROUND_DOWN 2dp ทุกเดือน — เมื่อหารไม่ลงตัว เดือนสุดท้ายจะค้างเศษเล็กน้อย (0.01–0.99 ฿) → NBV ไม่ถึง residualValue ที่กำหนด → งบดุลผิดเล็กน้อยตลอดอายุสินทรัพย์

แก้: ใช้ tx.depreciationEntry.count({ where: { assetId } }) ตรวจว่าเป็นเดือนสุดท้าย — ถ้าใช่ post remainingBase แทน monthlyAmount ทำให้ Σ posts = depreciableBase พอดี

2.6 Other Income reversedBy link ไม่ทำงาน

บัค: ลิงก์ "ดูเอกสาร Reversing Entry" ไม่เคย render เพราะ field ไม่มีจริงใน DB

PR #808 · ไฟล์: other-income.service.ts + OtherIncomeViewPage.tsx

Frontend อ้าง doc.reversedById แต่ Prisma schema มีแค่ self-relation reversedBy (no scalar FK) → undefined ตลอด → ลิงก์ไม่เคยแสดง

แก้: Backend include: { reversedBy: { select: { id, docNumber } } } + Frontend ใช้ doc.reversedBy?.id

2.7 dailySheet timezone drift

บัค: ใบที่ออกหลัง 17:00 UTC (00:00 BKK) ตกใน daily sheet ของวันถัดไป

psql $DB_URL -f \
  apps/api/prisma/migrations-manual/\

  4. Verification queries: 
    -- 21-3105 ต้องมี rows หลัง migration
SELECT account_code, COUNT(*), SUM(credit_amount)
FROM journal_lines jl
JOIN journal_entries je ON je.id = jl.journal_entry_id
WHERE je.metadata->>'flow' = 'expense-payroll'
  AND jl.account_code IN ('21-1104', '21-3105')
GROUP BY account_code;

-- Expected: 21-3105 has rows, 21-1104 PAYROLL count = 0
  5. Smoke test: สร้างเอกสาร EX + PAYROLL test 1 ใบแต่ละชนิด ผ่าน UI → ดู JE Preview ครบบรรทัด → POST → ตรวจ Trial Balance ของ 21-3105/21-3106

12FAQ

Q1: ทำไมต้องแยก 21-3105/21-3106 จาก 21-1104?

เดิม 21-1104 ปนกัน 2 ความหมาย: AP จริง + SSO ค้างนำส่ง. แยกแล้วทำให้ Trial Balance ตรวจสอบง่าย + สปส.1-10 รายงานชัดเจน + auditor reconcile ได้

Q2: ถ้ามี JE เก่าที่ใช้ 11-2104 ผิด ต้องทำยังไง?

สร้าง adjusting JE manual: Dr 11-4101 / Cr 11-2104 ตามยอดที่ผิด — postedAt = วันที่ทำ adjustment, reference = "Reclass VAT input per audit 2026-05-11" — ทำก่อนยื่น ภ.พ.30 รอบถัดไป

Q3: V15 บล็อก ACCRUAL+WHT ใช่ไหม? ถ้าจริงๆ ต้องตั้งหนี้พร้อม WHT?

ไม่ใช่ — ตั้งหนี้ ACCRUAL ไม่ใส่ WHT ก่อน → เวลาจ่ายเงินจริง สร้าง VENDOR_SETTLEMENT ระบุ WHT ที่ขั้นนั้น. WHT ลงงวดเดือนที่จ่าย ตรงตาม ม.50

Q4: ค่าเสื่อม cron จะเริ่มทำงานเมื่อไหร่?

สิ้นเดือน 2026-05 (วันที่ 31 พ.ค.) 01:00 BKK — fix นี้ deploy ก่อนสิ้นเดือนแน่นอน. งวด มี.ค.-เม.ย. ที่หายไป ต้อง run manual ผ่าน runManual() endpoint

Q5: ทำไมต้องเพิ่ม account_role_map?

เพื่อให้ owner แก้ routing ของบัญชี ผ่าน admin UI โดยไม่ต้อง deploy code ใหม่ — เช่น เปลี่ยน wht_individual จาก 21-3102 → บัญชีอื่นในอนาคต. ปัจจุบัน payroll template เป็น POC ใช้ roles.code() แล้ว, templates อื่นจะ migrate ตามใน PR ถัดไป

Q6: PR #809 (weekly deps) เกี่ยวกับบัญชีไหม?

ไม่เกี่ยว — แค่อัปเดต 18 npm packages (patch/minor) ของ React, AWS SDK, Sentry, ฯลฯ + เอกสาร progress report. ไม่มี code change ในระบบบัญชี

Q7: หน้า /expenses/:id/voucher (PaymentVoucher) เปิดเมื่อไหร่?
