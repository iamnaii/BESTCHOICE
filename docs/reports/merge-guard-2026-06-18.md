   - `p.amountPaid` is a `Prisma.Decimal` field from the database; converting with `Number()` before comparison risks floating-point precision loss for amounts like `1999.99` (which may become `1999.9899999...`)
   - The `AMOUNT_TOLERANCE_BAHT` tolerance (likely 1฿) partially mitigates this but the root issue remains
   - Fix:
     ```ts
     // Replace:
     this.amountMatches(Number(p.amountPaid), line.amount)
     // With:
     this.amountMatches(p.amountPaid.toNumber(), line.amount)  // or use Decimal comparison
     ```
   - Note: While `.toNumber()` has the same risk, the proper fix is to compare using `Decimal` arithmetic:
     ```ts
     private amountMatches(a: Prisma.Decimal, b: number): boolean {
       return a.sub(new Prisma.Decimal(b)).abs().lte(BankReconciliationService.AMOUNT_TOLERANCE_BAHT);
     }
     ```
   - File: `apps/api/src/modules/accounting/bank-reconciliation.service.ts`

2. **`two-factor.controller.ts` — `@UseGuards(JwtAuthGuard)` without `RolesGuard`**

   - Recommendation: Add a comment on the controller explaining this is intentional (to prevent future code reviewers from flagging it):

