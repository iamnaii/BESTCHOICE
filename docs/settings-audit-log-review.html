📌 อ่านก่อนตรวจ — audit log คืออะไรในบริบทนี้

ทุกครั้งที่มีคน แก้ไขค่าตั้งค่าของระบบ (เช่น อัตราค่าปรับ, อัตรา VAT, prefix เลขที่เอกสาร, ผู้ดูแลเงินสดย่อย, การจับคู่บัญชีกับ role) ระบบจะเขียน "บันทึกร่องรอย" 1 แถวลงตาราง audit_logs โดยอัตโนมัติ — ระบุ ใคร / ทำอะไร / เมื่อไหร่ / ค่าเดิม→ค่าใหม่ บันทึกนี้แก้ย้อนหลังไม่ได้ (immutable) และมีการเซ็นด้วยลายเซ็นดิจิทัลแบบลูกโซ่ (hash chain) เพื่อกันการแก้ไขหลักฐาน

1เหตุการณ์ที่ถูกบันทึก (Audit Actions)

โมดูล Settings เขียน audit log ใน 5 กลุ่มเหตุการณ์ ทั้งหมดเก็บ ค่าเดิม และ ค่าใหม่ เพื่อให้ฝ่ายบัญชีเทียบได้ว่าเปลี่ยนอะไรไป:

รหัสเหตุการณ์ (action) เกิดเมื่อ บันทึกอะไร ความสำคัญต่อบัญชี
SYSTEM_CONFIG_UPDATE
SYSTEM_CONFIG_CREATE		 แก้ไข/สร้างค่าตั้งค่าใดๆ เช่น อัตราค่าปรับ, VAT_RATE, prefix เอกสาร, โหมด maker-checker ชื่อ key + ค่าเดิม → ค่าใหม่ สูง — ค่าเหล่านี้กระทบการคำนวณเงิน/ภาษีโดยตรง
ROLE_MAP_CREATED
ROLE_MAP_UPDATED
ROLE_MAP_DEACTIVATED		 เพิ่ม/แก้/ปิด การจับคู่ระหว่าง role กับผังบัญชี (account role map) role, รหัสบัญชี + สรุปความเปลี่ยนแปลงภาษาไทย (diffSummary) สูง — กำหนดว่าธุรกรรมลงบัญชีตัวไหน
PETTY_CASH_CUSTODIAN_ASSIGNED เปลี่ยนตัวผู้ดูแลเงินสดย่อยของบริษัท รหัสบริษัท + ผู้ดูแลคนเดิม → คนใหม่ กลาง — segregation of duties เงินสดย่อย
DOC_SEQUENCE_RESET รีเซ็ต/ตรวจลำดับเลขที่เอกสาร ประเภทเอกสาร, งวด, เลขล่าสุดต่อประเภท (ไว้ diagnostic) กลาง — ความต่อเนื่องเลขที่เอกสารภาษี
EXPORT_BLOCKED มีคนพยายามส่งออกข้อมูลตอนที่ระบบปิดการส่งออก ผู้ใช้, เส้นทาง (route), IP, เหตุผลที่ถูกบล็อก กลาง — PDPA / ควบคุมการนำข้อมูลออก
✅ จุดที่ตรวจได้ง่าย

แต่ละแถวมี ค่าเดิมและค่าใหม่คู่กัน — ฝ่ายบัญชีไม่ต้องเดาว่า "ก่อนหน้านี้ตั้งไว้เท่าไร" ตัวอย่าง: ถ้าอัตราค่าปรับต่อวันถูกแก้จาก 20 เป็น 50 บาท log จะเก็บ oldValue: {value:"20"} และ newValue: {value:"50"} พร้อม userId ของคนแก้

2กลไกคุ้มกันหลักฐาน 4 ชั้น

สิ่งที่ทำให้ audit log นี้ "เชื่อถือได้" ในเชิงตรวจสอบ ไม่ใช่แค่ log ธรรมดา:

ชั้นที่ 1

ปิดบังค่าลับ (Redact)

ค่าที่เป็นความลับ (token, password, api key, credential) จะถูกแทนด้วย [REDACTED] ก่อนบันทึก — log รู้ว่า "key นี้ถูกแก้" แต่ไม่เก็บค่าจริง ป้องกันความลับรั่วผ่านหน้าตรวจสอบ

settings.constants.ts → redact()
ชั้นที่ 2

ต้องมีตัวตนผู้ทำ

ถ้าไม่มี userId (เช่น งานอัตโนมัติของระบบ) จะข้ามการเขียน log โดยตั้งใจ — ทุกแถวใน audit log จึงผูกกับ "คนจริง" ที่รับผิดชอบเสมอ

audit.service.ts → if (!userId) return
ชั้นที่ 3

Audit ล้ม ไม่ล้มธุรกรรม

ถ้าเขียน log ไม่สำเร็จ ระบบจะไม่ย้อนการแก้ค่าตั้งค่า แต่จะแจ้งเตือนทีมเทคนิคทันที (Sentry) — กันไม่ให้มี "รูโหว่" ในลูกโซ่หลักฐานแบบเงียบๆ

Sentry alert on audit write fail
ชั้นที่ 4

ลูกโซ่กันแก้ไข (Hash Chain)

ทุกแถวถูกเซ็นด้วย SHA-256 ที่อ้างอิงแถวก่อนหน้า ถ้าใครแก้ข้อมูลเก่าในฐานข้อมูลตรงๆ ระบบตรวจสอบจะจับได้ทันทีว่าลูกโซ่ขาดตรงไหน

verifyChain() → tamper-evident
⚠️ ข้อควรรู้สำหรับผู้ตรวจ

มีเหตุการณ์เดียว (EXPORT_BLOCKED) ที่เขียน log แบบตรง ไม่ผ่านลูกโซ่ hash chain เพราะเกิดขึ้นในชั้นป้องกันก่อนถึงระบบหลัก — ยังบันทึกครบ (ใคร/IP/เหตุผล) แต่ไม่ได้อยู่ในลูกโซ่กันแก้ไข ส่วนอีก 4 กลุ่มเหตุการณ์อยู่ในลูกโซ่ทั้งหมด

3เส้นทางการบันทึก (สำหรับเข้าใจภาพรวม)

ผู้ใช้กดบันทึกค่าตั้งค่า (เช่น แก้อัตราค่าปรับ) │ ▼ ตรวจสิทธิ์ (role) ──► ไม่มีสิทธิ์ ──► ปฏิเสธ (403) │ ▼ ตรวจความถูกต้องของค่า ──► ค่าไม่ผ่าน ──► ปฏิเสธ + ข้อความไทย │ ▼ บันทึกค่าใหม่ลงฐานข้อมูล (มีการดึง "ค่าเดิม" มาเก็บก่อน) │ ▼ เขียน AUDIT LOG → ใคร · action · ค่าเดิม→ค่าใหม่ · เวลา · hash │ ▼ ปรากฏในหน้า /audit-logs ให้ตรวจย้อนหลัง (กรองตามคน/ประเภท/ช่วงเวลา)

4รายการที่ฝ่ายบัญชีควรตรวจ (Checklist)

แนะนำให้ใช้เป็นแนวทางตรวจสอบเป็นงวด (รายเดือน/รายไตรมาส) ผ่านหน้า /audit-logs:

  • การเปลี่ยนค่าที่กระทบเงิน/ภาษี มีเหตุผลรองรับ กรอง action = SYSTEM_CONFIG_UPDATE ดูรายการที่แตะ อัตราค่าปรับ (late_fee_*), VAT_RATE, การจับคู่บัญชี — ทุกการเปลี่ยนต้องมีผู้อนุมัติ/เอกสารประกอบ
  • คนที่แก้มีสิทธิ์ถูกต้อง ตรวจว่าผู้ทำรายการ (userId) เป็น OWNER / FINANCE_MANAGER / ACCOUNTANT ตามนโยบาย ไม่ใช่ role ที่ไม่ควรแตะการตั้งค่าการเงิน
  • การเปลี่ยนผู้ดูแลเงินสดย่อย สมเหตุสมผล กรอง PETTY_CASH_CUSTODIAN_ASSIGNED — ยืนยันว่าการโยกผู้ดูแลตรงกับการเปลี่ยนตัวจริงในองค์กร (segregation of duties)
  • ลำดับเลขที่เอกสารไม่ถูกรีเซ็ตโดยพลการ กรอง DOC_SEQUENCE_RESET — การรีเซ็ตเลขเอกสารภาษีควรมีน้อยและมีเหตุผล เพราะกระทบความต่อเนื่องเลขที่
  • ความพยายามส่งออกข้อมูลที่ถูกบล็อก กรอง EXPORT_BLOCKED — ดูว่ามีการพยายามดึงข้อมูลออกผิดปกติหรือไม่ (PDPA)
  • ความสมบูรณ์ของลูกโซ่หลักฐาน ขอทีมเทคนิครัน "ตรวจลูกโซ่" (verifyChain) เป็นงวด — ผลต้องเป็น "ลูกโซ่สมบูรณ์" ถ้าพบจุดขาด แปลว่ามีการแก้ไขฐานข้อมูลนอกระบบ ต้องสอบสวน
💡 ข้อสรุปสำหรับผู้บริหาร/ผู้ตรวจ

การตั้งค่าที่กระทบตัวเลขทางบัญชี ทุกครั้งถูกบันทึกว่าใครแก้ แก้จากอะไรเป็นอะไร และเมื่อไหร่ — พร้อมกลไกกันแก้ไขหลักฐานย้อนหลัง ฝ่ายบัญชีสามารถตรวจย้อนหลังได้เต็มรูปแบบผ่านหน้า /audit-logs โดยกรองตามคน ประเภทเหตุการณ์ และช่วงเวลา
