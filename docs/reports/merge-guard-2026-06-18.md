#### вЪ†пЄП Warning вАФ `Number()` on Prisma Decimal in financial reduce (2 occurrences)

**File:** `apps/api/src/modules/accounting/receivables-report.service.ts`

```ts
// WARNING: native JS number summation loses Decimal precision
const total = lines.reduce((sum, l) => sum + Number(l.debit ?? 0), 0);
// ...
amount: Number(l.debit ?? 0),
```

Native `Number()` in a `reduce` accumulator accumulates floating-point rounding error across many journal lines. Per project rules, Prisma Decimal fields must stay as `Prisma.Decimal` for financial calculations. This is in the bad-debt report context so it affects display totals, not double-entry posting.

**Fix:** Replace with:
```ts
import { Decimal } from '@prisma/client/runtime/library';
const total = lines.reduce((sum, l) => sum.plus(new Decimal(l.debit?.toString() ?? '0')), new Decimal(0));
amount: new Decimal(l.debit?.toString() ?? '0'),
```

#### вЪ†пЄП Warning вАФ `sale.findMany()` missing `deletedAt: null` (2 occurrences)

**File:** `apps/api/src/modules/accounting/transactional-report.service.ts` (probable)

Two `sale.findMany()` calls in P&L/cost report context are missing `deletedAt: null`:

```ts
// MISSING deletedAt: null
this.prisma.sale.findMany({
  where: { createdAt: dateRange, ...branchFilter },
  select: { saleType: true, netAmount: true, downPaymentAmount: true, createdAt: true },
})
// MISSING deletedAt: null

  where: { createdAt: dateRange, ...branchFilter },
  select: { createdAt: true, product: { select: { costPrice: true } } },

```

These could include soft-deleted sales in financial reports, distorting revenue and COGS figures.

**Fix:** Add `deletedAt: null` to both `where` clauses.

The `contract` relation has `deletedAt: null`, but the `payment` record itself does not. Soft-deleted payment records (if any exist) would appear in this query.
