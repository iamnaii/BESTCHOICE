```typescript

    rate1Down: Number(map.get('sticker.rate1.defaultDown') ?? 0),   // ⚠️
```

`rate1Down` and `rate2Down` are down-payment defaults used for sticker display only — they drive price labels on printed stickers, not financial transaction records. The `StickerData` / `StickerRate` interfaces use `number` (not `Decimal`), consistent with display-only intent. Since no Decimal arithmetic or DB writes occur with these values, the precision risk is low. However, the coding convention discourages `Number()` on any monetary value.

Suggestion: either use `.toNumber()` with a comment (`// display only`) or define a typed `Decimal`-based path if these values ever feed a calculation.

**2. Batch endpoint uses comma-separated query param without rate-limiting path**

However, there is no per-user throttle override for this potentially expensive batch query (each ID triggers a full product + pricing-template lookup chain). Consider adding a `@Throttle()` decorator or explicit limit.

**3. `StickerData` interface declared in `stickers.service.ts` (not in `packages/shared/`)**  
The interface is used in both service and potentially the frontend. If the web app reconstructs the type manually, it risks drift. Moving to `packages/shared/types/` would be cleaner long-term.

**4. Route collision risk: `GET products/data` vs `GET product/:productId/data`**  
NestJS routes are matched in declaration order. The new `GET products/data` must be declared **before** `GET product/:productId/data` in the controller to avoid Express treating `products` as a `productId` param. Verify the declaration order is correct.
