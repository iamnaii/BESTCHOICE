#### âš ď¸Ź WARNING â€” N+1 Query on `findAll()`

**File:** `apps/api/src/modules/customers/customers.service.ts` (~line 141)

`findAll()` calls `tierService.getCustomerTier()` for every customer in the paginated result (up to 50 per page). Each call issues **2 DB queries** (contract `findMany` + repossession `count`). With page limit 50, this becomes **100 extra DB queries per list request** â€” a classic N+1.

```ts
const withTier = await Promise.all(
  enriched.map(async (c) => {
    try {
      const t = await this.tierService.getCustomerTier(c.id);
      return { ...c, tier: t.tier };
    } catch {
      return { ...c, tier: 'NEW' as const };
    }
  }),
);
```

The in-memory comment acknowledges "valid for small shops", but 100 queries/request will slow the customers list page significantly as data grows.

**Recommended fix:** Batch all contract + repossession data in a single query (or two queries with `customerId IN [...]`) and compute tiers in memory, rather than calling `getCustomerTier()` per customer.

#### đźš¨ CRITICAL â€” `JwtAudienceGuard` guard ordering: audience check is always a no-op

**Files:** `apps/api/src/app.module.ts`, `apps/api/src/modules/auth/guards/jwt-audience.guard.ts`

The guard is registered as `APP_GUARD`:

```ts
// app.module.ts
{
  provide: APP_GUARD,
  useClass: JwtAudienceGuard,
},
```

The comment claims *"JwtAudienceGuard runs globally after per-controller JwtAuthGuard sets req.user"* â€” **this is incorrect**. In NestJS, global guards (registered via `APP_GUARD`) run **before** per-controller/per-route guards like `@UseGuards(JwtAuthGuard)`. The execution order is:

```
Global guards (ThrottlerGuard â†’ CsrfGuard â†’ JwtAudienceGuard)
  â†’ Controller guards (JwtAuthGuard â†’ RolesGuard)
```

This means when `JwtAudienceGuard.canActivate()` runs, `req.user` is **always `undefined`** on any endpoint protected by `@UseGuards(JwtAuthGuard)` (because JwtAuthGuard has not yet executed). The guard's fallback `if (!req.user) return true` causes it to pass every request through, making the entire audience enforcement a no-op.

**Impact:** A shop-audience JWT (e.g., from a LINE-logged-in customer) can access admin endpoints like `GET /api/customers`, `GET /api/contracts`, etc. The security boundary this feature was designed to create does not exist at runtime, despite unit tests passing (tests call the guard directly with a pre-populated `req.user`).

**Recommended fix (two options):**

*Option A â€” Decode JWT in the guard itself (recommended):*
Inject `JwtService` and read the raw `Authorization` header to extract/verify the `aud` claim independently, without relying on `req.user`.

```ts
constructor(private reflector: Reflector, private jwtService: JwtService) {}

canActivate(ctx: ExecutionContext): boolean {
  const req = ctx.switchToHttp().getRequest();
  const token = req.headers.authorization?.replace('Bearer ', '');
  if (!token) return true; // no JWT â†’ defer to JwtAuthGuard
  const payload = this.jwtService.decode(token) as any;
  const aud = payload?.aud;
  // ... enforce aud
}
```

*Option B â€” Apply as per-controller guard after JwtAuthGuard:*
Remove from `APP_GUARD` and require controllers to declare `@UseGuards(JwtAuthGuard, JwtAudienceGuard)` explicitly.

**Fix:**

```ts

```
