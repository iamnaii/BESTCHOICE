**W1 — Monetary DTO fields typed as `number` (float precision risk)**  
Files: `apps/api/src/modules/overdue/dto/partial-payment-reschedule.dto.ts:95`, `dto/log-contact.dto.ts:47,57`

```typescript
// partial-payment-reschedule.dto.ts
@Type(() => Number)
@IsNumber({ maxDecimalPlaces: 2 })
amountPaid!: number;  // ← should be string
```

Service converts immediately via `new Prisma.Decimal(dto.amountPaid)` where `dto.amountPaid` is a JS `number`. While `@IsNumber({ maxDecimalPlaces: 2 })` limits the input, some 2-decimal JS floats (e.g., `0.30`) are not exactly representable in IEEE 754 and may produce imprecise Decimal values. The established pattern in this codebase is to use `Prisma.Decimal` from DB reads — DTOs should accept `string` + `@IsDecimal()` / `@Matches(/^\d+(\.\d{1,2})?$/)` to eliminate all risk.

**Applies also to** `LogContactDto.settlementAmount` and `secondSettlementAmount`.

**Fix**: Change `amountPaid!: number` → `amountPaid!: string`, validate with `@IsDecimal({ decimal_digits: '0,2' })`, and pass the string directly to `new Prisma.Decimal(dto.amountPaid)`. Same for `settlementAmount` fields.

**W2 — New controller methods lack explicit `@Roles` on `ESCALATION_LEGAL` path — SoD enforced only in service**  
File: `apps/api/src/modules/overdue/overdue.controller.ts:291`

```typescript
@Post(':contractId/escalate')
@Roles('OWNER', 'BRANCH_MANAGER', 'FINANCE_MANAGER', 'SALES')
escalate(...)
```

The `LEGAL` escalation action has Separation-of-Duties gating *inside* `overdue.service.ts`. While this is documented in the code comment, SoD logic in a service (not enforced by a guard) can be bypassed if a future refactor moves the method or re-uses the service elsewhere. Consider extracting `LEGAL` into its own endpoint with `@Roles('OWNER', 'FINANCE_MANAGER')` at the controller level, or add a note to `.claude/rules/security.md`.
