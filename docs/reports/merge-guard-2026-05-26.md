### ⚠️ W1 — Plain-text string in `password` field of system user

**File:** `apps/api/src/modules/staff-chat/services/canned-response-sender.service.ts:530`

```ts
create: {
  email: 'system@bestchoice.internal',
  password: 'NEVER_LOGIN_SYSTEM_USER',   // ← plain text, not bcrypt hash
  ...
  isActive: false,
}
```

The `User.password` column normally stores bcrypt hashes. Storing a raw ASCII string here
is a code smell: it deviates from the column's invariant, and any future code that iterates
User rows and assumes `password` is a valid bcrypt string (e.g. audit tools, migration scripts)
could behave unexpectedly.

Since `isActive: false`, the auth guard rejects login before bcrypt comparison — the immediate
security risk is low. However, the correct fix is to store a bcrypt hash of a random string:

```ts
password: await bcrypt.hash(randomBytes(32).toString('hex'), 10),
```

Or, if `isActive: false` users are always blocked before password check, use a sentinel clearly
outside bcrypt hash space (e.g. `'!SYSTEM!'`) with an invariant comment.
