### Issues

#### WARNING — Missing error handling on async `getConfig` calls
**File**: `facebook-webhook.controller.ts`  
**Methods**: `verifyWebhook`, `verifySignature`, `handleDataDeletion`

All three now `await this.integrationConfig.getConfig('facebook')` without a try/catch. If the DB is unavailable at webhook verification time (e.g., during initial FB webhook setup), NestJS will return a 500 instead of a controlled 400. For the `verifyWebhook` GET endpoint, a 500 prevents the FB app from completing webhook registration. The two POST paths (HMAC check + data deletion) are less critical since FB retries non-2xx.

**Suggested fix**:
```ts
private async getAppSecret(): Promise<string | undefined> {
  try {
    const cfg = await this.integrationConfig.getConfig('facebook');
    return cfg.appSecret || undefined;
  } catch {
    return this.configService.get<string>('FB_APP_SECRET');
  }
}
```

#### INFO — Env fallback not visible in controller diff
**File**: `facebook-webhook.controller.ts`  
The comment in the controller's JSDoc says "FB_APP_SECRET env as fallback" but the controller only calls `this.integrationConfig.getConfig('facebook')`. The fallback must live inside `IntegrationConfigService.getConfig`. Worth confirming that service implements the fallback, otherwise loss of DB connectivity = complete loss of FB webhook functionality.

### Verdict: **APPROVE**
Functionally sound. One warning that is advisable to address before production load, but does not block merge.

### Issues

#### WARNING — Fragile `useEffect` dependency on `onCountsChange`
**File**: `BubbleList.tsx` — `useEffect(() => { ... }, [allBubbles, onCountsChange])`

`onCountsChange` is a callback prop in the dependency array. Today the only call site passes `setBubbleCounts` (a `useState` setter — stable by guarantee), so there is no bug. However, if the component is later reused with a non-memoized inline callback, every render will re-run the effect, which calls `onCountsChange`, which sets state, which triggers another render — infinite loop.

**Suggested fix**: Wrap in `useCallback` at the call site, or accept a `useRef` for the callback inside `BubbleList`:
```ts
// In TemplateEditorPane.tsx — already safe since setBubbleCounts is stable,
// but document the contract explicitly:
const stableCounts = useCallback((c) => setBubbleCounts(c), []);
// ...
<BubbleList onCountsChange={stableCounts} ... />
```
Or add an ESLint comment at the call site to document the stability assumption.

#### INFO — Implicit `allBubbles` sort-order assumption
**File**: `bubble-reorder-logic.ts`

`reorderBubbles` operates on `allBubbles` in the order returned by the API query. The logic is correct only if the API returns bubbles sorted by `sortOrder`. This appears to be the existing contract, but it is not asserted in tests. Consider adding a test case that verifies behavior when bubbles arrive out of `sortOrder` sequence.

### Test coverage

### Verdict: **APPROVE**
