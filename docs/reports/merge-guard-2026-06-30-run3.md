#### Warning — `rejectQC` persists no AuditLog with rejection reason or actor
**File:** `apps/api/src/modules/purchase-orders/services/po-receiving.service.ts`  
**Method:** `rejectQC()`

The `reason` string is validated, returned in the API response body, but never written to the `AuditLog` table. The global `AuditInterceptor` will record the HTTP event (endpoint hit, userId, timestamp), but it does not capture:
- Which product IDs were rejected
- The rejection reason text
- The per-product names

For a soft-delete that permanently removes inventory from the QC queue, this gap means there is no queryable audit trail answering "why was product X removed and who approved it." The CLAUDE.md v5 note on `MDM_AUTO_LOCK` and the v3 Cascade→Restrict hardening both show the project treats inventory lifecycle events as audit-worthy.

**Suggested fix:**
```typescript
await tx.auditLog.create({
  data: {
    userId,          // pass userId into rejectQC(productIds, reason, userId)
    action: 'QC_REJECTED',
    entity: 'product',
    entityId: productIds.join(','),
    newValue: { productIds, reason, count: productIds.length },
  },
});
```
This requires adding `userId` as a parameter to `rejectQC()` and wiring it through controller → service → receiving service.

#### Info — `rejectQCDto` missing `userId` propagation path
The controller calls `this.purchaseOrdersService.rejectQC(dto.productIds, dto.reason)` without passing `@CurrentUser()`. If the audit log fix above is implemented, the controller signature must also change to inject `@CurrentUser() user: { id: string }`.

### Recommendation: ⚠️ REVIEW
Audit log for rejection reason is a meaningful traceability gap for inventory. Should fix before merge to main.
