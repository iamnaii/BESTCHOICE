### Fix A — ApproverSection shape mismatch (`ApproverSection.tsx`)

**Problem**: The old code called `api.get('/users?roles=OWNER,FINANCE_MANAGER,ACCOUNTANT')` and treated the response as a bare `UserRow[]`. The `/users` endpoint returns a paginated envelope `{ data, total, page, limit }`, not a bare array — causing `r?.map is not a function` at runtime.

**Fix**:
```ts
const APPROVER_ROLES = ['OWNER', 'FINANCE_MANAGER', 'ACCOUNTANT'];
// ...
queryFn: async () => {
  const res = await api.get('/users?limit=200');
  const list: UserRow[] = res.data?.data ?? (Array.isArray(res.data) ? res.data : []);
  return list.filter((u) => APPROVER_ROLES.includes(u.role));
},
```

Correct fix. The fallback `Array.isArray(res.data) ? res.data : []` guards against future API shape changes. `limit=200` is safe for this role set (OWNER + FINANCE_MANAGER + ACCOUNTANT users will never reach 200 in a typical deployment).

### Fix B — Modal scroll layout (`ExpenseFormV4.tsx`)

**Problem**: Sticky header/footer approach broke on viewport heights < modal height — the sticky positioning was relative to the scrolling ancestor, causing header/footer to scroll out of view.

**Fix**: Converted to flex-column layout:
```
outer: fixed inset-0 flex items-center justify-center p-4
inner: max-h-[95vh] flex flex-col
header: flex-none
content: flex-1 overflow-y-auto
footer: flex-none
```

Correct CSS fix. Header and footer now stay pinned; only the content area scrolls.
