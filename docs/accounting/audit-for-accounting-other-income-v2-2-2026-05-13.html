1Override JV — แก้รายการบัญชีเองก่อน POSTใหม่

ปกติระบบจะ generate รายการบัญชี (JE) อัตโนมัติจากข้อมูลในเอกสาร — แต่ในบางกรณีฝ่ายบัญชีจำเป็นต้องแก้รายการเองก่อน POST (เช่น แยกบัญชีย่อย, จัดทำ JE พิเศษ, ปรับ contra-account)

ขั้นตอนการใช้งาน

1กรอกเอกสารตามปกติ — ระบบ generate JE อัตโนมัติให้ดู
2กด checkbox "ใช้เอง (Override)"
ระบบจะแสดงคำเตือน + ให้กด checkbox ยืนยันว่ารับผิดชอบความถูกต้อง
3แก้ไขตาราง JE ได้
เพิ่ม/ลบ/แก้บรรทัด ระบบตรวจสด V1/V2/V5 ทันที
4กด POST
ระบบ validate อีกครั้ง + บันทึก audit log (original vs modified)
5เอกสารแสดง ✏ marker
หน้ารายการแสดง icon ดินสอข้างหมายเลข เพื่อให้สังเกตว่าเอกสารนี้ถูก override

กฎ Validation (V1/V2/V5) ที่ระบบบังคับ

กฎคำอธิบายข้อความ error (ไทย)
V1ยอด Dr รวม = ยอด Cr รวม (tolerance ±0.01 บาท)Dr (X) ≠ Cr (Y) — ผลต่าง Z บาท
V2ต้องมีอย่างน้อย 2 บรรทัดต้องมีอย่างน้อย 2 บรรทัด
V5แต่ละบรรทัดต้องเป็น Dr หรือ Cr ไม่ใช่ทั้งคู่บรรทัด XX-XXXX มีทั้ง Dr และ Cr / ไม่มีทั้ง Dr และ Cr

ตัวอย่าง Audit Log เมื่อ Override

{ "action": "JV_OVERRIDDEN", "entity": "other_income", "entityId": "OI-20260513-0042", "userId": "USER-001", "oldValue": { // JE ที่ระบบ generate "jvLines": [ { "accountCode": "11-1201", "debit": "1000.00", "credit": "0" }, { "accountCode": "42-1102", "debit": "0", "credit": "1000.00" } ] }, "newValue": { // JE หลังถูก override "jvLines": [ { "accountCode": "11-1201", "debit": "1500.00", "credit": "0" }, { "accountCode": "42-1102", "debit": "0", "credit": "1500.00" } ], "diffSummary": "แก้ Dr 11-1201 จาก 1,000.00 → 1,500.00; แก้ Cr 42-1102 จาก 1,000.00 → 1,500.00" } }

หมายเหตุสำคัญ: เอกสารที่ถูก override จะมี flag isOverridden=true ใน DB ตลอดไป แม้จะ reverse แล้วก็ตาม — ใช้เป็นหลักฐานในการ audit ได้

สิ่งที่ฝ่ายบัญชีต้องตรวจ (Override JV)

  • ทดสอบกด Override — ใส่ Dr=1000 / Cr=999 ระบบควรกัน POST + ขึ้น error V1
  • ทดสอบลบบรรทัดให้เหลือ 1 — ระบบควรกัน POST + ขึ้น error V2
  • ทดสอบใส่ทั้ง Dr+Cr ในบรรทัดเดียว — ระบบควรกัน POST + ขึ้น error V5
  • POST สำเร็จ → เปิด View page — ต้องเห็น JV_OVERRIDDEN ใน audit log พร้อมสรุป diff ภาษาไทย
  • เปิด List page — เอกสารที่ override ต้องมีไอคอน ✏ ข้างหมายเลข
  • ทดสอบกด Override + ยกเลิก (ไม่ติ๊ก checkbox ยืนยัน) — ระบบไม่ควรเข้า override mode

2Maker-Checker Toggle — เปิด/ปิดระบบอนุมัติใหม่

ปกติฝ่ายบัญชีจะกำหนดให้ คนสร้างเอกสาร ≠ คนอนุมัติ (segregation of duties) — แต่บางช่วงต้องการความเร็ว เช่น ช่วงปิดบัญชี/ทำ adjustment เร่งด่วน — สามารถปิดได้ที่หน้า Settings (เฉพาะ OWNER)

วิธีใช้

  1. เปิด /settings#users (Tab "ผู้ใช้งาน")
  2. กด Switch "ระบบ Maker-Checker"
  3. ระบบแสดง confirmation dialog แตกต่างตามทิศทาง:

    เปิดระบบ (OFF → ON)

    • เอกสารทุกฉบับต้องผ่านผู้อนุมัติก่อน POST
    • เอกสาร DRAFT ปัจจุบันต้องส่งอนุมัติ
    • ผู้สร้าง ≠ ผู้อนุมัติ

    ปิดระบบ (ON → OFF)

    • เอกสาร READY จะ auto-approve ทันที
    • เอกสารใหม่ POST ทันที (ไม่ต้องอนุมัติ)
    • ระบบแสดงจำนวนเอกสาร READY ตอนนี้: N ฉบับ
  4. ติ๊ก checkbox "ฉันเข้าใจและยืนยัน..."
  5. กด "ยืนยันเปิด" / "ยืนยันปิด"

กฎสิทธิ์: ผู้ใช้ที่ไม่ใช่ OWNER เห็น Toggle แต่กดไม่ได้ (disabled) พร้อมข้อความ "เฉพาะ OWNER เท่านั้นที่เปลี่ยนได้"

3Reopen Period — เปิดงวดที่ปิดแล้วใหม่

เมื่อปิดงวดบัญชีไปแล้วแต่ต้องการเปิดใหม่ชั่วคราว (เช่น พบเอกสารผิด, ลืมบันทึก, auditor ขอแก้) — ระบบมี SOP บังคับใน UI ให้บันทึกเหตุผล + ภ.พ.30 ยื่นหรือยัง + audit log

Modal บังคับใส่เหตุผล

เมื่อกด เปิดงวด ที่ปิดแล้ว ระบบจะแสดง modal ที่กรอกครบ 3 ส่วนถึงจะส่งได้:

ฟิลด์ประเภทข้อมูล
reasonType Radio (เลือก 1) WRONG_ENTRY — พบเอกสารผิดต้อง reverse
MISSED_RECORD — ลืมบันทึกรายการสำคัญ
AUDITOR_REQUEST — แก้ไขตามคำขอ auditor
OTHER — อื่นๆ (ระบุในบันทึก)
reasonFree text (≥ 10 ตัวอักษร)บันทึกรายละเอียดของเหตุผล (เช่น "เอกสาร OI-20260415-0042 ระบุลูกค้าผิด ต้อง reverse")
taxFiledRadio (ใช่/ยังไม่)ภ.พ.30 งวดนี้ยื่นแล้วหรือยัง — ถ้ายื่นแล้วต้องยื่นแก้ไข

Banner เตือนบนหน้า List

ระหว่างที่งวดเปิดอยู่ — ทุกหน้า /other-income และ /expenses จะแสดง Banner สีเหลือง:

  • กรอกไม่ครบ — ปุ่ม "ยืนยันเปิดงวด" disabled

  • ผู้ใช้ที่ไม่ใช่ OWNER กดเปิดงวด — ระบบ block (403 Forbidden)

สิ่งที่ฝ่ายบัญชีต้องตรวจ (Pagination)

  • กด Next — URL เปลี่ยนเป็น ?page=2
  • เปลี่ยน size เป็น 100 — URL update + page reset ไป 1
  • พิมพ์ "ไปหน้า 5" + Enter — ไปหน้า 5 ทันที
  • Refresh browser — อยู่หน้าเดิม (URL state)
  • กรอง filter (เช่น status) — page reset ไป 1 อัตโนมัติ
  • เปลี่ยน filter เป็น READY — เรียงตามเก่าก่อน (ผู้รอ approve นานสุดอยู่บนสุด)

AAudit Log — รายการที่ระบบจะบันทึก

การกระทำที่มีผลต่อระบบบัญชีในเวอร์ชัน v2.2 ที่ระบบจะ บันทึก audit log อัตโนมัติทุกครั้ง (immutable + hash-chained):

Action Stringเมื่อไรEntityข้อมูลที่บันทึก
JV_OVERRIDDEN POST เอกสาร Other Income ด้วย override other_income oldValue = JE auto / newValue = JE override + diffSummary ภาษาไทย
CONFIG_CHANGED OWNER toggle Maker-Checker system_config oldValue = ค่าเดิม / newValue = ค่าใหม่
PERIOD_REOPENED OWNER เปิดงวดที่ปิดแล้ว accounting_period newValue = { reasonType, reason, taxFiled, reopenedAt }
PERIOD_CLOSED OWNER/ACCOUNTANT ปิดงวด accounting_period newValue = { closedAt }

Audit log Properties (ทุก record): userId (FK ไป User), action (string), entity (lowercase), entityId, oldValue (JSON), newValue (JSON), ipAddress, createdAt, rowHash (sha256 chain)

✓Checklist สุดท้าย — ฝ่ายบัญชี Sign-off

ก่อนกด "อนุมัติให้ go-live" ฝ่ายบัญชีกรุณาตรวจรายการต่อไปนี้ครบทุกข้อ:

ก. การทดสอบ Workflow (Manual UAT)

  • Override JV — ทดสอบ V1/V2/V5 ครบ 3 errors + POST สำเร็จ 1 เคส + ตรวจ ✏ marker + ตรวจ audit log diff
  • Maker-Checker — OWNER toggle ON→OFF + ตรวจจำนวน READY ที่แสดง + ตรวจ audit log
  • Reopen Period — เปิดงวดด้วยเหตุผล 4 ประเภท + ตรวจ Banner + ตรวจ audit log + ปิดงวดอีกครั้ง
  • Settings 5 Tabs — เปิดทุก tab + แก้ VAT rate + แก้ attachment threshold + ทดสอบ /accounting/periods redirect
  • Pagination — ทดสอบ Next/Prev/Jump-to-page + เปลี่ยน size + refresh ยังอยู่หน้าเดิม

ข. การตรวจสอบความปลอดภัย

  • ผู้ใช้ไม่ใช่ OWNER — เข้า /settings ไม่ได้ (redirect ไปหน้าหลัก)
  • ผู้ใช้ไม่ใช่ OWNER — กด Maker-Checker Toggle ไม่ได้ (disabled)
  • ผู้ใช้ไม่ใช่ OWNER — เปิดงวด (POST /expenses/periods/reopen) ไม่ได้ (403)
  • Override JV requires confirmation — ติ๊ก checkbox "ฉันเข้าใจ..." ก่อนถึงจะกด "เปิดโหมดแก้ไข" ได้

ค. การตรวจสอบ Audit Trail

  • JV_OVERRIDDEN audit — มี oldValue (auto JE) + newValue (override + diff)
  • CONFIG_CHANGED audit — มี oldValue + newValue ของ Maker-Checker flag
  • PERIOD_REOPENED audit — มี reasonType + reason + taxFiled + reopenedAt
  • PERIOD_CLOSED audit — มี closedAt
  • ทุก audit เก็บ userId จริงและ ipAddress — ไม่ใช่ NULL หรือ "system"

ง. การตรวจสอบ Data Integrity

  • Override JV เอกสาร — ค่า isOverridden ใน DB = true (persist ถาวร)
  • Reopened Period — ค่า reopenReason + taxFiled ใน DB ครบ
  • Migration applied — รัน prisma migrate deploy สำเร็จ (3 migrations: pagination indexes, reopen metadata, seed PR3 keys)
  • SystemConfig seed — มี 4 keys ใหม่ใน DB (VAT_RATE, VAT_PRICE_TYPE_DEFAULT, ATTACHMENT_REQUIRED_ABOVE_AMOUNT, ATTACHMENT_ALLOWED_TYPES)
