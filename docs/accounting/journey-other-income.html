2วงจรชีวิตเอกสาร

DRAFT → POSTED ↘ REVERSED
StatusความหมายJE สร้างแล้ว?แก้ไขได้?ลบได้?
DRAFTร่าง — ยังไม่ post — ไม่มีผลทางบัญชีไม่มีแก้ไขได้soft-delete ได้
POSTEDโพสต์แล้ว + ออกเลข receipt (RT-YYYYMM-NNNNN)มี — Auto JEไม่ได้ห้าม — ใช้ reverse
REVERSEDกลับรายการแล้ว — มี -R doc + reversal JEมี 2 JE (ต้นฉบับ + reversal)ไม่ไม่ — สถานะ final

กฎการเปลี่ยนสถานะ

  • DRAFT → POSTED: ผ่าน POST /other-income/:id/post — สร้าง JE + ออก receiptNo. ต้องผ่าน V1-V10 หมด
  • POSTED → REVERSED: ผ่าน POST /other-income/:id/reverse — สร้าง doc ใหม่ที่ docNumber + '-R', มี reversesId ชี้กลับ + JE สลับ Dr/Cr
  • DRAFT → ลบ: ผ่าน DELETE /other-income/:id — soft-delete (set deletedAt)
  • POSTED → ลบ: ห้าม — backend throws ConflictException
  • Concurrent post protection: ใช้ pg_advisory_xact_lock ป้องกัน race

ใช้เมื่อพบว่า POST เอกสารผิด (ลูกค้าผิด, ยอดผิด, บัญชีผิด ฯลฯ). ห้าม edit หรือ hard-delete — ใช้ reverse

วิธีทำใน UI

  1. เข้า /other-income/:id (view page)
  2. กดปุ่ม "Reverse" — เลือก reverseReason: INPUT_ERROR / CUSTOMER_REQUEST / DUPLICATE / WRONG_ACCOUNT / WRONG_AMOUNT / OTHER
  3. กรอก reverseNote (optional)
  4. ยืนยัน → ระบบสร้าง doc ใหม่ {docNumber}-R + JE สลับ Dr/Cr
  5. เอกสารต้นฉบับ status → REVERSED, doc ใหม่ status = POSTED

JE Reversal Pattern

JE ใหม่ใช้ postedAt = วันที่ยกเลิก (BKK noon) — ไม่ใช่วันที่ต้นฉบับ. ทุกบรรทัดสลับ Dr/Cr

ระวัง: Reverse ทำให้รายงานเดือนเปลี่ยน! ถ้าต้นฉบับเป็นเดือน พ.ค. และ reverse ในเดือน มิ.ย. → P/L ทั้ง 2 เดือนกระทบ

12Daily Sheet — สรุปรายวัน

เข้าผ่าน /other-income/daily-sheet หรือ click card "สรุปรายวัน" จาก list page

โครงสร้างหน้า

  • Summary: รายได้ก่อนภาษีรวม · VAT รวม · WHT รวม · สุทธิรับรวม · จำนวนเอกสาร
  • By Account: แยกตาม 42-XXXX — 42-1102 รวม X฿, 42-1105 รวม Y฿
  • By Payment: แยกตามช่องทางรับเงิน — 11-1101 รวม Z฿ ...
  • Doc list: POSTED docs ของวันนั้น (เลขเอกสาร, ลูกค้า, ยอด)

การใช้ปิดเงินสดประจำวัน

  1. ดูยอด "By Payment" — แต่ละช่องทางควรตรงกับเงินสดในมือ/ยอดธนาคาร
  2. ถ้าไม่ตรง → ตรวจ docs ของวันนั้น ดูว่ามีคน reverse ภายหลังหรือไม่ (REVERSED ไม่ขึ้น Daily Sheet)
  3. ปริ้นต์/save Daily Sheet เป็นหลักฐานการปิดวัน

13Receipt — ใบเสร็จ

เลข receipt RT-YYYYMM-NNNNN ออกอัตโนมัติเมื่อ POST — เข้าผ่าน /other-income/:id/receipt เพื่อปริ้นต์

Fieldหมายเหตุ
receiptNoauto-generated ที่ post time, immutable
customerNoteข้อความที่กรอกใน Section 4 — แสดงในใบเสร็จ
postedAtวันที่ใน timestamp (ใบเสร็จจะ format เป็นวันที่ไทย)
items[]list รายการที่ขึ้นใบเสร็จ

14เช็คลิสต์รายวัน/รายเดือนสำหรับฝ่ายบัญชี

รายวัน

  • เข้า list page → เช็ค DRAFT count — ถ้าค้างเยอะ → ตามนักการเงินให้ post/ลบ
  • เข้า Daily Sheet วันก่อนหน้า → ดูยอด "By Payment" ตรงกับยอดเงินสด/ธนาคารที่ปิดวัน
  • ถ้ามี dispute จากลูกค้า (วันนั้น) → ตรวจ docs + เตรียม reverse ถ้าจำเป็น

รายเดือน

  • Trial Balance (TB) → ดูบัญชี 42-XXXX มี balance ตรงกับ P/L หรือไม่
  • VAT report → 21-2101 movement = ยอด VAT จากใบกำกับ 42-1105 (ถ้ามี)
  • WHT certificate → ออกใบ ภงด.50/51 ให้ผู้จ่าย (ปกติ KBank/SCB ออกให้เราเอง)
  • ตรวจ docs REVERSED ของเดือน — เก็บเอกสารหลักฐาน (reverseReason + reverseNote)

15ข้อผิดพลาดที่พบบ่อย + วิธีแก้

อาการสาเหตุวิธีแก้
POST ไม่ได้ — ปุ่ม disabled มี validation error อยู่ในกล่องแดง อ่าน "ต้องแก้ไข N ข้อก่อน POST" → แก้ทีละข้อ. ระหว่างนี้ "บันทึกร่าง" ใช้ได้
กรอกบัญชี 41-1101 / 53-XXXX ไม่ใช่บัญชีกลุ่ม 42-XXXX เปลี่ยนเป็น 42-XXXX. ถ้าเป็นรายได้หลัก (HP interest) → ใช้ flow 2A/2B แทน
ยอดรับจริงต่างจากยอดสุทธิ <= 1฿ เศษสตางค์ UI จะแสดง AdjustmentTable ให้เลือก 53-1503 (รับเกิน) หรือ 52-1104 (รับขาด, ต้องมี approver)
ยอดรับจริงต่างจากยอดสุทธิ > 1฿ ผิดพลาดมาก กลับไปเช็คตัวเลขใน Section 2 ก่อน. ถ้ายังต้องรับยอดไม่ตรง → ใส่ adjustment + note อธิบาย
กดบันทึก & POST แล้วเด้ง error "ต้องแนบไฟล์" ยอดรับ ≥ ATTACHMENT_THRESHOLD แต่ไม่ได้แนบ กลับไปบันทึกร่างก่อน → แนบไฟล์ใน Section 8 → POST
POST แล้วพบบัญชี/ยอดผิด — ห้าม edit/delete — ใช้ reverse แล้ว post ใหม่
กรอก 42-1103 ใน dropdown ไม่ขึ้น BLOCKED ใน UI (auto-posted ผ่าน 2B) ค่าปรับล่าช้าจะ post อัตโนมัติตอนรับเงินค่างวด ไม่ต้องคีย์ในโมดูลนี้

16FAQ บัญชี

Q: ดอกเบี้ยเงินฝาก KBank — ใช้ "ลูกค้า" ฟิลด์ใส่อะไร?

A: ใช้ free-text "KBank" (พิมพ์ในช่อง customer search โดยไม่เลือกจาก dropdown). ระบบจะเก็บเป็น counterpartyName ไม่ใช่ customerId FK

Q: WHT 15% ที่ KBank หักจากเรา — ขอคืนได้?

A: ได้. ระบบ Dr 11-4103 (ภาษีหัก ณ ที่จ่ายรอเรียกคืน) เก็บไว้. ปลายปียื่นภาษีนิติบุคคล → ขอคืนภาษี (refund) หรือเครดิตภาษีปีถัดไป

Q: VAT จากกำไรขายสินทรัพย์ — นำส่งได้รึยัง?

A: ระบบ Cr 21-2101 (VAT settled) ทันทีตอน POST — รวมใน ภ.พ.30 เดือนนั้น. ฝ่ายบัญชีต้องนำส่ง VAT รายเดือนตามจริง

Q: มีรายได้อื่นที่อยากบันทึก แต่ไม่มีบัญชี 42-XXXX ที่ตรง?

A: ปรึกษานักบัญชีก่อน — อาจต้องเพิ่ม account ใหม่ในผัง. ห้ามใส่ 41-XXXX (รายได้หลัก) เด็ดขาด เพราะจะปนกับยอดสัญญาผ่อน

Q: ทำไมหน้านี้ไม่มี "รออนุมัติ" (READY)?

A: Backend ปัจจุบันมี 3 สถานะเท่านั้น: DRAFT / POSTED / REVERSED. รายได้อื่นยอดน้อย — Owner approve ตอน POST อยู่แล้ว ไม่จำเป็นต้องมี middle state. ถ้าอยากเพิ่ม approval workflow → ต้องทำ feature ใหม่ (schema migration + endpoint + role + UI)

Q: ผู้บันทึก + ผู้อนุมัติ ใน Section 7 ทำไมเป็นคนเดียวกัน?

A: ตามต้นแบบ owner ตอนนี้ — auto-fill จาก current user. ถ้าจะแยก dual-sign approval ต้องเพิ่ม approverId field ใน schema + workflow ใหม่. รอ Phase ถัดไป

Q: Reverse แล้วเอกสารต้นฉบับยังขึ้นในรายงาน?

A: ใช่ — status เป็น REVERSED (ไม่ใช่ลบ). รายงาน P/L หัก reversal ออกอัตโนมัติเพราะ JE มี 2 entries (ต้นฉบับ + reversal สลับ DR/CR). Daily Sheet กรอง REVERSED ออก แต่ trial balance ยังเห็น
