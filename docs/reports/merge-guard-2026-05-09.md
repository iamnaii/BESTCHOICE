#### ⚠️ Warning — `Number()` on Prisma Decimal in payment recording path

**File**: `apps/api/src/modules/paysolutions/paysolutions.service.ts`

```ts
// PartialPaymentLink.amount is Decimal @db.Decimal(12,2) in Prisma
await this.paymentsService.recordPayment(
  payment.contractId,
  payment.installmentNo,
  Number(link.amount),   // ← Decimal → JS number conversion
  'ONLINE_GATEWAY',
  ...
);
```

`link.amount` is fetched from `PartialPaymentLink` (Prisma `Decimal`). It is converted to `number` before passing to `recordPayment()`. The existing `recordPayment` signature accepts `amount: number`, so the conversion is required by the callee's API. However, this conversion can lose sub-cent precision on edge cases and violates the project convention of keeping Decimal throughout financial flows.

**Root cause**: `PaymentsService.recordPayment()` was designed with `amount: number` before the Decimal-everywhere rule was enforced (see hardening v2/v4 Decimal fixes). The webhook path now surfaces this pre-existing gap.

**Risk level**: Low-to-medium. Thai baht installment amounts are `Decimal(12,2)` — at most 2 decimal places — so JS float representation is exact for values under 2^53 / 100 (≈ 90 trillion baht). However the architectural debt should be tracked.

**Recommended fix**: Change `PaymentsService.recordPayment(amount: number, ...)` to `amount: Prisma.Decimal | number` (or `Decimal`) and pass `link.amount` directly without `Number()`. This is a one-line change to the signature + removing the `Number()` cast.
