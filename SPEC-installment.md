## 1. Executive Summary

ระบบผ่อนชำระ (Installment Management System) สำหรับเครือร้านจำหน่ายมือถือ มากกว่า 5 สาขา เพื่อ **แทนที่ระบบเดิมทั้งหมด** รองรับการขายผ่อนชำระสินค้าทุกประเภท (มือถือใหม่, มือสอง, แท็บเล็ต, อุปกรณ์เสริม) พร้อมระบบติดตามหนี้, แจ้งเตือนอัตโนมัติ, และ dashboard สำหรับผู้บริหาร

### เป้าหมายหลัก

- แทนที่ระบบเดิมด้วยระบบที่รวมศูนย์ข้อมูลทุกสาขา
- ลดงาน manual ในการติดตามการชำระเงินและจัดทำรายงาน
- เพิ่มความโปร่งใสและควบคุมความเสี่ยงด้านลูกหนี้
- รองรับการใช้งานบนทุกอุปกรณ์ (Responsive)

## 3. แผนผ่อนชำระ (Installment Plans)

### 3.1 ประเภทแผนผ่อน

ระบบรองรับ 3 ประเภท:

| ประเภท | คำอธิบาย | ดอกเบี้ย | หมายเหตุ |
|---|---|---|---|
| **ผ่อนตรงกับร้าน** | ร้านจัดสินเชื่อเอง ไม่ผ่านธนาคาร | คิดจากลูกค้า | แผนหลักที่ระบบต้องจัดการเต็มรูปแบบ |
| **ผ่อนผ่านบัตรเครดิต/ไฟแนนซ์** | ลูกค้าใช้บัตรเครดิตหรือไฟแนนซ์ภายนอก | ตามเงื่อนไขผู้ออกบัตร | ระบบบันทึกเป็นรายการขายผ่อน แต่ไม่ต้องจัดการงวด |
| **ผ่อนพร้อมดอกเบี้ย** | คิดดอกเบี้ยจากลูกค้าโดยตรง | Flat rate | เป็น sub-type ของผ่อนตรงกับร้าน |

### 3.2 เงื่อนไขการผ่อน

| พารามิเตอร์ | ค่าเริ่มต้น | ปรับได้? | หมายเหตุ |
|---|---|---|---|
| **ระยะเวลาผ่อน** | 6–12 เดือน | ✅ | Admin กำหนดตัวเลือกที่ให้พนักงานขายเลือก |
| **เงินดาวน์ขั้นต่ำ** | 15% ของราคาสินค้า | ✅ | ปรับได้ภายหลังโดย Admin |
| **อัตราดอกเบี้ย** | 8% ต่อเดือน (Flat rate) | ✅ | ปรับได้ภายหลัง — ดูหมายเหตุด้านล่าง |
| **ค่าปรับจ่ายช้า** | 100 บาท/วัน | ✅ | สูงสุดไม่เกิน 200 บาท ต่อ 1 งวดที่ค้าง |
| **ปิดบัญชีก่อนกำหนด** | อนุญาต | — | ลดดอกเบี้ย 50% ของงวดที่เหลือ |

> ⚠️ **หมายเหตุเรื่องดอกเบี้ย:** อัตรา 8% ต่อเดือนแบบ Flat rate เป็นอัตราที่สูงมาก (เทียบเท่า ~96% ต่อปี) กรุณายืนยันว่าเป็น:
> - **(A)** 8% ต่อเดือน flat rate จริง (เช่น ผ่อน 10,000 บาท 6 เดือน = ดอกเบี้ย 4,800 บาท) → **ยอมรับความเสี่ยงทางกฎหมาย** เนื่องจากเกินเพดาน พ.ร.บ. อัตราดอกเบี้ย
> - **(B)** 8% flat rate **ต่อสัญญา** (ไม่ใช่ต่อเดือน) → เช่น ผ่อน 10,000 บาท = ดอกเบี้ยรวม 800 บาท
> - **(C)** อัตราอื่น → โปรดระบุ
>
> ระบบจะออกแบบให้ **configurable** ทั้งหมด ไม่ว่ากรณีใด

### 3.3 สูตรการคำนวณ (Flat Rate)

```
ดอกเบี้ยรวม      = ราคาสินค้า × อัตราดอกเบี้ย × จำนวนเดือน
ยอมผ่อนรวม       = (ราคาสินค้า - เงินดาวน์) + ดอกเบี้ยรวม
ค่างวดต่อเดือน    = ยอดผ่อนรวม ÷ จำนวนเดือน
```

### 3.4 การปิดบัญชีก่อนกำหนด

```
ดอกเบี้ยที่เหลือ         = ค่างวดดอกเบี้ย × จำนวนงวดที่เหลือ
ส่วนลด                   = ดอกเบี้ยที่เหลือ × 50%
ยอมปิดบัญชี              = เงินต้นคงเหลือ + (ดอกเบี้ยที่เหลือ - ส่วนลด)
```

### 3.5 ค่าปรับจ่ายล่าช้า

```
ค่าปรับ = MIN(จำนวนวันที่เกิน × 100, 200) ต่องวด
```

- เริ่มคิดค่าปรับหลังวันครบกำหนดชำระ 1 วัน
- ค่าปรับสูงสุด 200 บาท ต่อ 1 งวดที่ค้าง (cap ที่ 2 วัน)
- ถ้าค้างหลายงวด คิดค่าปรับแยกแต่ละงวด

## 4. ข้อมูลลูกค้าและเอกสาร

### 4.1 ข้อมูลที่ต้องเก็บ

**ข้อมูลส่วนตัว:**
- ชื่อ-นามสกุล
- เลขบัตรประชาชน (13 หลัก)
- ที่อยู่ตามบัตร / ที่อยู่ปัจจุบัน
- เบอร์โทรศัพท์ (หลัก + สำรอง)
- LINE ID (สำหรับแจ้งเตือน)
- อาชีพ / สถานที่ทำงาน

**เอกสารแนบ (upload):**
- สำเนาบัตรประชาชน (ภาพถ่าย/สแกน)
- สลิปเงินเดือน หรือ หลักฐานรายได้

### 4.2 หลักประกัน

- **ใช้ตัวเครื่องเป็นหลักประกัน** — ระบบต้องผูก IMEI/Serial Number ของเครื่องกับสัญญาผ่อน
- กรณีเปลี่ยนเครื่อง → ต้องอัพเดท IMEI/Serial ในสัญญาใหม่

## 5. Lifecycle ของสัญญาผ่อน

### 5.1 สถานะสัญญา (Contract Status)

```
[DRAFT] → [ACTIVE] → [COMPLETED]
              ↓
         [OVERDUE] → [DEFAULT] → [CLOSED_BAD_DEBT]
              ↓
         [EARLY_PAYOFF]

         [ACTIVE] → [EXCHANGED] (เปลี่ยนเครื่อง → สัญญาใหม่)
```

| สถานะ | คำอธิบาย |
|---|---|
| `DRAFT` | สร้างสัญญาแล้ว รอวางดาวน์ / ยืนยัน |
| `ACTIVE` | สัญญาเริ่มต้น กำลังผ่อนชำระปกติ |
| `OVERDUE` | มีงวดที่เลยกำหนดชำระ (ยังไม่ default) |
| `DEFAULT` | ค้างชำระเกินเกณฑ์ที่กำหนด (2 งวดติดต่อกัน) |
| `EARLY_PAYOFF` | ลูกค้าจ่ายปิดก่อนกำหนด |
| `COMPLETED` | ชำระครบทุกงวดแล้ว |
| `EXCHANGED` | สัญญาถูกยกเลิกเนื่องจากเปลี่ยนเครื่อง (link ไปสัญญาใหม่) |
| `CLOSED_BAD_DEBT` | ตัดเป็นหนี้สูญ |

### 5.2 Flow การเปลี่ยนเครื่องระหว่างผ่อน

1. คำนวณยอมคงค้างของสัญญาเดิม (เงินต้น + ดอกเบี้ยค้าง + ค่าปรับถ้ามี)
2. กำหนดราคาเครื่องใหม่
3. ลูกค้าจ่ายส่วนต่าง (ถ้าเครื่องใหม่ราคาสูงกว่า) หรือได้เครดิตคืน
4. ปิดสัญญาเดิม → สถานะ `EXCHANGED`
5. สร้างสัญญาใหม่ พร้อม link กลับไปสัญญาเดิม

| ช่องทาง | รายละเอียด | การยืนยัน |

## 7. ระบบแจ้งเตือน (Notifications)

### 7.1 ช่องทาง

- **LINE** (หลัก) — ผ่าน LINE Official Account + Messaging API
- **SMS** (เสริม) — สำหรับลูกค้าที่ไม่มี LINE หรือ LINE ส่งไม่สำเร็จ

### 7.2 เหตุการณ์ที่แจ้งเตือน

| เหตุการณ์ | เมื่อไหร่ | ผู้รับ | ช่องทาง |
|---|---|---|---|
| เตือนก่อนครบกำหนด | 3 วัน + 1 วันก่อน | ลูกค้า | LINE → fallback SMS |
| เลยกำหนดชำระ | วันถัดไป, 3 วัน, 7 วัน | ลูกค้า | LINE + SMS |
| ชำระสำเร็จ | ทันที | ลูกค้า | LINE |
| สัญญาเข้าสถานะ OVERDUE | ทันที | ผู้จัดการสาขา | ในระบบ + LINE |
| สัญญาเข้าสถานะ DEFAULT | ทันที | เจ้าของ/ผู้บริหาร | ในระบบ + LINE |

### 7.3 Template ข้อความ

- ข้อความทุกประเภทต้อง **configurable** โดย Admin
- รองรับ placeholder: `{ชื่อลูกค้า}`, `{เลขสัญญา}`, `{จำนวนเงิน}`, `{วันครบกำหนด}` ฯลฯ

```

เจ้าของ       → เห็นทุกอย่าง ทุกสาขา + ปรับ config ได้
```

## 9. UI/UX Requirements

### 9.1 แพลตฟอร์ม

- **Web Application** — Responsive Design (Desktop, Tablet, Mobile)
- Mobile-first สำหรับพนักงานขายหน้าร้าน (ใช้มือถือ/แท็บเล็ตเป็นหลัก)
- Desktop-first สำหรับ Dashboard/รายงาน (เจ้าของ/ฝ่ายบัญชี)

### 9.2 หน้าจอหลัก

**A. พนักงานขาย:**
1. **หน้าสร้างสัญญา** — Form wizard แบบ step-by-step:
   - Step 1: เลือกสินค้า (ค้นหาจากชื่อ/รุ่น/IMEI)
   - Step 2: กรอกข้อมูลลูกค้า + upload เอกสาร
   - Step 3: เลือกแผนผ่อน (ดาวน์, จำนวนงวด) → แสดง preview ตารางผ่อน
   - Step 4: Preview สัญญา → ลูกค้าเซ็นบนหน้าจอ (e-Signature) → จัดเก็บ e-Document
2. **หน้ารายการสัญญา** — ดูสัญญาทั้งหมดในสาขา, filter ตามสถานะ
3. **หน้าบันทึกการชำระ** — เลือกสัญญา → บันทึกงวด + แนบหลักฐาน
4. **หน้าแจ้งเตือน** — รายการแจ้งเตือนส่วนตัว (สัญญาใกล้ครบกำหนด, ค้างชำระ)

**B. ผู้จัดการสาขา:**
5. **Dashboard สาขา** — สรุปยอดลูกหนี้, จำนวนสัญญาตามสถานะ, แจ้งเตือน
6. **หน้าจัดการสต็อก** — ดูสินค้าในสาขา, นำเข้า, โอนระหว่างสาขา
7. **หน้าติดตามหนี้** — รายชื่อลูกค้าค้างชำระ + บันทึกการติดตาม (call log)

**C. เจ้าของ/ผู้บริหาร:**
8. **Dashboard รวม** — ยอดรวมทุกสาขา, กราฟแนวโน้ม, KPIs
9. **รายงาน** — (รายละเอียดใน Section 10)
10. **หน้า Settings** — ปรับ config ดอกเบี้ย, ดาวน์ขั้นต่ำ, ค่าปรับ, template แจ้งเตือน
11. **หน้ายึดเครื่อง/Resell** — บันทึกการยึดคืน, ตีราคา, ตั้งราคาขายต่อ
12. **หน้าสต็อกรวม** — ดูสต็อกทุกสาขา, สินค้ายึดคืน, สินค้ารอ resell

**D. ฝ่ายบัญชี/การเงิน:**
13. **หน้ารายการชำระเงิน** — ดูทุกรายการชำระจากทุกสาขา, reconcile
14. **รายงานการเงิน** — รายได้, กำไรขาดทุน, aging report

### 9.3 UX Guidelines

- ใช้ภาษาไทยทั้งหมด
- สร้างสัญญาใหม่ต้องทำเสร็จภายใน **5 นาที** หรือน้อยกว่า
- บันทึกการชำระต้องทำได้ภายใน **30 วินาที**
- รองรับการค้นหาลูกค้าด้วย: ชื่อ, เบอร์โทร, เลขบัตร ปชช., เลขสัญญา
- แสดง **color coding** ตามสถานะสัญญา (เขียว=ปกติ, เหลือง=ใกล้ครบกำหนด, แดง=ค้างชำระ)

## 10. Reports & Dashboard

### 10.1 Dashboard (Real-time)

- **ยอดลูกหนี้รวม** ทุกสาขา (เงินต้นคงเหลือ + ดอกเบี้ยค้างรับ)
- **จำนวนสัญญา** แยกตามสถานะ (Active, Overdue, Default, Completed)
- **กราฟแนวโน้ม** ยอมผ่อนใหม่ vs ยอดชำระ รายเดือน
- **อัตราการค้างชำระ** (% ของสัญญาที่ overdue)
- **Top overdue** — 10 อันดับลูกค้าที่ค้างชำระมากที่สุด
- Filter ได้ตาม: สาขา, ช่วงเวลา

### 10.2 รายงาน (Export PDF/Excel)

| รายงาน | รายละเอียด | ผู้ใช้หลัก |
|---|---|---|
| **รายได้/กำไรขาดทุน** | รายได้จากดอกเบี้ย, ค่าปรับ, หนี้สูญ | เจ้าของ, ฝ่ายบัญชี |
| **Aging Report** | จัดกลุ่มลูกหนี้ตามอายุหนี้ (1-30, 31-60, 61-90, 90+ วัน) | เจ้าของ, ฝ่ายบัญชี |
| **ลูกค้าค้างชำระ/เสี่ยงสูง** | รายชื่อ + ยอดค้าง + จำนวนวันที่ค้าง | ผู้จัดการ, เจ้าของ |
| **เปรียบเทียบพนักงานขาย** | ยอดขายผ่อน, จำนวนสัญญา, อัตรา overdue | ผู้จัดการ, เจ้าของ |
| **เปรียบเทียบสาขา** | ยอดรวมแต่ละสาขา, performance metrics | เจ้าของ |
| **รายงานการชำระประจำวัน** | สรุปรายการชำระทั้งวัน แยกช่องทาง | ฝ่ายบัญชี |
| **รายงานสต็อกสินค้า** | คงเหลือแยกสาขา, สินค้าค้างสต็อก, สินค้ายึดคืน | ผู้จัดการ, เจ้าของ |
| **รายงานเครื่องยึดคืน/Resell** | รายการยึดคืน, ราคาตี, ค่าซ่อม, กำไร/ขาดทุนจาก resell | เจ้าของ, ฝ่ายบัญชี |
| **รายงานการสั่งซื้อ (PO)** | สรุป PO ตามสถานะ, ยอดซื้อตาม supplier, ของที่ยังไม่ได้รับ | ผู้จัดการ, เจ้าของ |
| **รายงาน Supplier** | ประวัติการซื้อแต่ละ supplier, เปรียบเทียบราคาทุน | เจ้าของ |
| **รายงานผลตรวจเช็ค** | สรุป grade distribution, อัตราผ่าน/ไม่ผ่าน, เครื่องที่ reject | ผู้จัดการ, เจ้าของ |

## 11. Edge Cases & Business Rules

### 11.1 การจ่ายล่าช้า

| สถานการณ์ | การจัดการ |
|---|---|
| ค้าง 1 วัน | เริ่มคิดค่าปรับ 100 บาท/วัน (cap 200 บาท/งวด) + แจ้งเตือนอัตโนมัติ |
| ค้าง 7 วัน+ | สถานะเปลี่ยนเป็น `OVERDUE` + แจ้ง ผจก.สาขา |
| ค้าง 2 งวดติดต่อกัน | สถานะเปลี่ยนเป็น `DEFAULT` + แจ้งเจ้าของ (ระบบเสนอให้โทรทวงเอง) |
| ตัดหนี้สูญ | เจ้าของ/ผู้บริหาร approve เท่านั้น → สถานะ `CLOSED_BAD_DEBT` |

> **หมายเหตุ:** ระบบ **ไม่** ส่งทีมทวงหนี้อัตโนมัติ — เจ้าของ/ผู้จัดการ ดำเนินการโทรทวงเอง ระบบเป็นเพียงเครื่องมือบันทึก call log และติดตามสถานะ

### 11.2 การจ่ายบางส่วน

- ลูกค้าจ่ายน้อยกว่ายอดงวด → ระบบบันทึกเป็น **partial payment**
- ยอดที่ขาด → ยกไปรวมกับงวดถัดไป
- ค่าปรับยังคงคำนวณจากยอดที่ค้างจ่าย

### 11.3 การจ่ายเกิน

- ลูกค้าจ่ายมากกว่ายอดงวด → ส่วนเกินหักเป็น **advance payment** ของงวดถัดไป
- กรณีจ่ายเกินงวดสุดท้าย → แจ้งพนักงานให้คืนเงินส่วนเกิน

### 11.4 เปลี่ยนเครื่องระหว่างผ่อน

- อนุญาตเฉพาะเมื่อ ผจก.สาขา หรือ เจ้าของ approve
- ลูกค้าต้องจ่ายยอดคงค้างทั้งหมดของเครื่องเดิม + ส่วนต่างราคาเครื่องใหม่
- สัญญาเดิม → `EXCHANGED` พร้อม reference ไปสัญญาใหม่
- สัญญาใหม่ → ระบุ origin = "exchanged from {เลขสัญญาเดิม}"

### 11.5 สินค้าถูกคืน/ยึดคืน

- กรณี default → เจ้าของตัดสินใจยึดเครื่องคืน
- ระบบบันทึกสถานะเครื่องว่าถูก "repossessed"
- คำนวณยอดขาดทุน (ยอดค้าง - มูลค่าเครื่องที่ยึดคืน)

### 11.6 ลูกค้าซ้ำ

- ลูกค้า 1 คน สามารถมีหลายสัญญาพร้อมกันได้
- ระบบแสดงประวัติทุกสัญญาของลูกค้า (รวมสัญญาที่ปิดแล้ว)
- **Risk flag:** ถ้าลูกค้ามีสัญญาที่ `OVERDUE` หรือ `DEFAULT` อยู่ → แสดงคำเตือนเมื่อสร้างสัญญาใหม่

## 13. Technical Recommendations

### 13.1 Tech Stack (แนะนำ)

| Layer | เทคโนโลยี | เหตุผล |
|---|---|---|
| **Frontend** | React + TypeScript + Tailwind CSS | Responsive, component-based, community ใหญ่ |
| **Backend** | Node.js (NestJS) หรือ Laravel (PHP) | NestJS: type-safe, scalable / Laravel: ถ้าทีมถนัด PHP |
| **Database** | PostgreSQL | รองรับ transaction, JSON fields, mature |
| **Cache** | Redis | สำหรับ session, notification queue |
| **Notification** | LINE Messaging API + SMS provider (ThaiBulkSMS / Twilio) | ตามที่ระบุ |
| **File Storage** | S3-compatible (MinIO for on-premise / AWS S3 for cloud) | เก็บเอกสาร, สลิป, หลักฐาน |
| **Report** | Server-side PDF generation + Excel export | ใช้ lib เช่น Puppeteer (PDF), ExcelJS |

### 13.2 Deployment (แนะนำ)

**แนะนำ: Cloud (AWS หรือ DigitalOcean)**

| ทางเลือก | ข้อดี | ข้อเสีย | ค่าใช้จ่ายโดยประมาณ |
|---|---|---|---|
| **DigitalOcean** | ง่าย, ราคาถูก, เหมาะกับ SME | จำกัด region ในเอเชีย | ~$40-80/เดือน |
| **AWS (Lightsail → EC2)** | ครบครัน, scale ได้, region สิงคโปร์ | ซับซ้อนกว่า, ค่าใช้จ่ายผันแปร | ~$50-150/เดือน |
| **On-premise** | ข้อมูลอยู่ในมือ, ไม่มี monthly cost | ต้องดูแลเอง, risk hardware fail, ไม่มี remote access | ค่า server ครั้งเดียว + ไฟ/เน็ต |

> **คำแนะนำ:** เริ่มจาก **DigitalOcean** หรือ **AWS Lightsail** เพราะง่ายต่อการเริ่มต้น, มี backup อัตโนมัติ, และรองรับการ scale เมื่อธุรกิจโต หากมีข้อกังวลเรื่องข้อมูลอยู่ในไทย สามารถใช้ AWS Singapore region ได้

### 13.3 Security

- Authentication: JWT + refresh token
- Password: bcrypt hashed
- HTTPS เท่านั้น
- Role-based access control (RBAC) ตาม Section 8
- Audit log ทุก action ที่เปลี่ยนแปลงข้อมูลสำคัญ
- เข้ารหัสข้อมูลส่วนบุคคล (เลขบัตร ปชช.) ใน database (encryption at rest)
- Backup อัตโนมัติทุกวัน + retention 30 วัน

## 14. Migration Plan

เนื่องจากเป็นการ **แทนที่ระบบเดิม** ควรมีแผน migration ดังนี้:

1. **Phase 1 — Data Export:** ดึงข้อมูลลูกค้า, สัญญาที่ยัง active, ประวัติการชำระ จากระบบเดิม
2. **Phase 2 — Data Mapping:** map ข้อมูลเข้า schema ใหม่ + ตรวจสอบความถูกต้อง
3. **Phase 3 — Parallel Run:** ใช้ระบบใหม่คู่ขนานกับระบบเดิม 1-2 สัปดาห์
4. **Phase 4 — Cutover:** ปิดระบบเดิม, ใช้ระบบใหม่เต็มรูปแบบ
5. **Phase 5 — Archive:** เก็บระบบเดิมไว้เป็น read-only อ้างอิง 6 เดือน

## 15. Risks & Tradeoffs

| ความเสี่ยง | ผลกระทบ | แนวทางแก้ไข |
|---|---|---|
| **อัตราดอกเบี้ยสูง (8%/เดือน)** | อาจขัด พ.ร.บ. ห้ามเรียกดอกเบี้ยเกินอัตรา (สูงสุด 15%/ปี สำหรับสินเชื่อส่วนบุคคล) | ปรึกษาทนายก่อน launch, ระบบต้อง configurable |
| **Manual payment verification** | พนักงานอาจลืมบันทึก, ข้อมูลไม่ real-time | ออกแบบ UX ให้บันทึกง่าย, มี daily reconciliation report |
| **LINE API dependency** | ลูกค้าต้อง add LINE OA, rate limit, ค่า messaging | มี SMS เป็น fallback, ตรวจสอบ delivery status |
| **Multi-branch data consistency** | ลูกค้าอาจไปจ่ายคนละสาขา | Centralized database, real-time sync |
| **Data migration จากระบบเดิม** | ข้อมูลอาจไม่สมบูรณ์หรือ format ต่างกัน | วางแผน mapping ละเอียด, มีระยะ parallel run |
| **พนักงานต่อต้านระบบใหม่** | ใช้งานไม่ถูกต้อง, ข้อมูลผิดพลาด | UX ที่ง่ายกว่าเดิม, training, onboarding plan |

## 17. ระบบเซ็นสัญญาออนไลน์ & E-Document

### 17.1 ภาพรวม

ระบบจัดทำสัญญาผ่อนชำระในรูปแบบ digital ทั้งหมด ลูกค้าเซ็นชื่อบนหน้าจอแท็บเล็ต/มือถือที่ร้าน แล้วจัดเก็บเป็น e-document ไม่ต้องพิมพ์กระดาษ

### 17.2 ขั้นตอนการเซ็นสัญญา

```
พนักงานสร้างสัญญา
    ↓
ระบบ generate สัญญา PDF (จาก template)
    ↓
แสดง preview สัญญาบนหน้าจอ (ลูกค้าอ่านทบทวน)
    ↓
ลูกค้าเซ็นลายเซ็นบนหน้าจอ (touch signature pad)
    ↓
พนักงาน/ร้านค้าเซ็นรับรอง
    ↓
ระบบฝังลายเซ็นลงใน PDF + ประทับ timestamp
    ↓
จัดเก็บ e-document + ส่งสำเนาให้ลูกค้าผ่าน LINE
```

### 17.3 Signature Pad

- ใช้ HTML5 Canvas สำหรับเซ็นลายเซ็นบนหน้าจอ (touch/stylus)
- รองรับทั้ง tablet และมือถือ
- สามารถ undo / clear แล้วเซ็นใหม่ได้
- บันทึกลายเซ็นเป็นรูปภาพ (PNG transparent background)
- เก็บ metadata: IP address, device info, timestamp (สำหรับ audit)

### 17.4 E-Document Management

**Template สัญญา:**
- Admin สร้าง/แก้ไข template สัญญาได้ (WYSIWYG editor หรือ upload template)
- รองรับ placeholder: `{ชื่อลูกค้า}`, `{เลขสัญญา}`, `{รายละเอียดสินค้า}`, `{ตารางผ่อน}`, `{ลายเซ็นลูกค้า}`, `{ลายเซ็นร้าน}` ฯลฯ
- Template แยกตามประเภท: สัญญาผ่อนตรง, สัญญาผ่อนบัตรเครดิต, สัญญาเปลี่ยนเครื่อง

**การจัดเก็บ:**
- เก็บ PDF ต้นฉบับ (signed) ใน file storage (S3/MinIO)
- เก็บ metadata ใน database: เลขสัญญา, วันที่เซ็น, ผู้เซ็น, hash ของไฟล์ (integrity check)
- **ห้ามแก้ไข** PDF หลังเซ็นแล้ว (immutable)
- Retention policy: เก็บอย่างน้อย 5 ปี (configurable)

**การเข้าถึง:**
- ค้นหาสัญญาด้วย: เลขสัญญา, ชื่อลูกค้า, วันที่
- ดู/ดาวน์โหลด PDF ได้ตาม role permission
- ส่งสำเนาให้ลูกค้าผ่าน LINE ได้ (ปุ่ม "ส่งสำเนาสัญญา")

### 17.5 เอกสารที่จัดเก็บในระบบ

| เอกสาร | สร้างเมื่อ |
|---|---|
| สัญญาผ่อนชำระ (signed PDF) | ตอนเปิดสัญญา |
| ใบเสร็จรับเงินดาวน์ | ตอนรับดาวน์ |
| ใบเสร็จค่างวด | ทุกครั้งที่ชำระ |
| ใบปิดบัญชี | ตอนปิดบัญชี |
| สำเนาบัตร ปชช. / สลิปเงินเดือน | ตอนสมัคร |
| หลักฐานการชำระ (สลิปโอน) | ทุกครั้งที่ชำระ |
| Purchase Order PDF | ตอนอนุมัติ PO |

> เอกสารทุกฉบับจัดเก็บถาวร เข้ารหัส และมี audit trail

## 18. ระบบปริ้นสติกเกอร์ติดเครื่องมือสอง

### 18.1 ภาพรวม

ปริ้นสติกเกอร์ QR/barcode สำหรับติดเครื่องมือถือมือสองทุกเครื่องที่รับเข้าสต็อก เพื่อให้สแกนดูข้อมูลสินค้าได้ทันที

### 18.2 Sticker Template System

Admin สร้าง/แก้ไข template สติกเกอร์ได้เอง (drag & drop editor หรือ predefined layouts) รองรับ **หลาย template** สำหรับใช้งานต่างสถานการณ์

**Placeholder ที่ใช้ได้:**

| Placeholder | ตัวอย่าง |
|---|---|
| `{product_code}` | SKU-20260001 |
| `{brand}` | Samsung |
| `{model}` | Galaxy S24 Ultra |
| `{imei}` | 356938035643809 |
| `{grade}` | A |
| `{selling_price}` | ฿12,900 |
| `{installment_price}` | ฿14,500 |
| `{branch}` | สาขาลาดพร้าว |
| `{date_received}` | 25/02/2026 |
| `{qr_code}` | QR ลิงก์ไปหน้ารายละเอียดสินค้าในระบบ |
| `{barcode}` | Barcode ของ product code |
| `{custom_field}` | ฟิลด์กำหนดเองได้ |

### 18.3 QR Code

- QR code ลิงก์ไปหน้ารายละเอียดสินค้าในระบบ (internal URL)
- พนักงานสแกนด้วยมือถือ → เปิดหน้าสินค้าทันที (ดูข้อมูล, ราคา, ผลตรวจเช็ค, ประวัติ)

### 18.4 การปริ้น

- รองรับเครื่องปริ้นสติกเกอร์ thermal label printer (เช่น Brother QL, Zebra, Xprinter)
- ขนาดสติกเกอร์ configurable (เช่น 50×30mm, 60×40mm, 62mm roll)
- ปริ้นทีละใบ หรือ batch print หลายใบพร้อมกัน

## 19. ระบบตรวจเช็คมือถือก่อนรับเข้าสต็อก (Phone Inspection)

### 19.1 ภาพรวม

ระบบ checklist สำหรับตรวจสภาพมือถือมือสองก่อนรับเข้าสต็อก เพื่อกำหนด condition grade และบันทึกสภาพเครื่องอย่างเป็นระบบ ทุกหัวข้อ **customize ได้ทั้งหมด** โดย Admin

### 19.2 Customizable Checklist

**Admin สามารถ:**
- สร้าง/แก้ไข/ลบ หัวข้อตรวจ (inspection item) ได้ไม่จำกัด
- จัดกลุ่มหัวข้อตรวจเป็นหมวดหมู่ (category)
- กำหนดประเภทการให้คะแนนแต่ละหัวข้อ: ผ่าน/ไม่ผ่าน, คะแนน 1-5, หรือ เลือก grade (A/B/C/D)
- กำหนดว่าหัวข้อไหน **บังคับ** vs **optional**
- กำหนด template checklist แยกตามประเภทเครื่อง (มือถือ vs แท็บเล็ต)
- เปิด/ปิดหัวข้อตรวจได้ตลอดเวลา

**ตัวอย่าง checklist เริ่มต้น (แก้ไขได้):**

| หมวด | หัวข้อตรวจ | ประเภท | บังคับ |
|---|---|---|---|
| **ภายนอก** | สภาพตัวเครื่อง (รอยขีดข่วน/บุบ) | Grade A-D | ✅ |
| | สภาพหน้าจอ (รอยร้าว/dead pixel) | Grade A-D | ✅ |
| | สภาพปุ่มกด (Power, Volume) | ผ่าน/ไม่ผ่าน | ✅ |
| | ช่องชาร์จ | ผ่าน/ไม่ผ่าน | ✅ |
| **การทำงาน** | หน้าจอสัมผัส (touch ทุกจุด) | ผ่าน/ไม่ผ่าน | ✅ |
| | ลำโพง/ไมค์ | ผ่าน/ไม่ผ่าน | ✅ |
| | กล้องหน้า/กล้องหลัง | ผ่าน/ไม่ผ่าน | ✅ |
| | Wi-Fi / Bluetooth | ผ่าน/ไม่ผ่าน | ✅ |
| | GPS / NFC | ผ่าน/ไม่ผ่าน | ❌ |
| | Face ID / สแกนนิ้ว | ผ่าน/ไม่ผ่าน | ✅ |
| **แบตเตอรี่** | สุขภาพแบตเตอรี่ (Battery Health %) | ตัวเลข | ✅ |

| **ซอฟต์แวร์** | รีเซ็ตเครื่องแล้ว | ผ่าน/ไม่ผ่าน | ✅ |
| | ปลดล็อค iCloud/Google Account | ผ่าน/ไม่ผ่าน | ✅ |
| | IMEI ไม่ถูก block | ผ่าน/ไม่ผ่าน | ✅ |

### 19.3 กระบวนการตรวจเช็ค

```
รับเครื่องมือสองเข้า (จาก supplier/trade-in/ยึดคืน)
    ↓
สร้างรายการสินค้าในระบบ (กรอก IMEI, ยี่ห้อ, รุ่น)
    ↓
เริ่มตรวจเช็ค → เปิด checklist (ตาม template ที่ตั้งไว้)
    ↓
พนักงานตรวจทีละหัวข้อ + ถ่ายรูปเครื่อง (อย่างน้อย 4 รูป: หน้า/หลัง/ซ้าย/ขวา)
    ↓
ระบบคำนวณ Overall Grade อัตโนมัติ (จาก weighted score)
    ↓
ผู้จัดการ review + ยืนยัน grade + กำหนดราคาขาย
    ↓
สินค้าเข้าสต็อก → พิมพ์สติกเกอร์ติดเครื่อง
```

### 19.4 Auto-Grading (configurable)

- Admin กำหนดเกณฑ์คะแนนสำหรับแต่ละ grade: เช่น A ≥ 90%, B ≥ 70%, C ≥ 50%, D < 50%
- ถ้ามีหัวข้อ critical ที่ "ไม่ผ่าน" → grade ไม่เกิน C อัตโนมัติ (configurable)
- ผู้จัดการสามารถ override grade ได้ (พร้อมเหตุผล)

### 19.5 ข้อมูลที่จัดเก็บ

- ผลตรวจทุกหัวข้อ + คะแนน
- ภาพถ่ายเครื่อง (ก่อนเข้าสต็อก)
- Overall grade (auto + manual override ถ้ามี)
- ผู้ตรวจ + วันเวลา
- หมายเหตุ/ข้อสังเกตพิเศษ
- ผลตรวจผูกกับตัวสินค้า → พนักงานขายดูได้ตอนแนะนำลูกค้า

## 20. ระบบ Supplier & หลายราคาขาย

### 20.1 Supplier Management

**ข้อมูล Supplier:**

| ฟิลด์ | บังคับ |
|---|---|
| ชื่อ Supplier / ชื่อบริษัท | ✅ |
| ชื่อผู้ติดต่อ | ✅ |
| เบอร์โทรศัพท์ (หลัก + สำรอง) | ✅ |
| LINE ID | ❌ |
| ที่อยู่ | ❌ |
| เลขประจำตัวผู้เสียภาษี | ❌ |
| หมายเหตุ | ❌ |
| สถานะ (Active / Inactive) | ✅ |

### 20.2 ประวัติการซื้อจาก Supplier

ทุกครั้งที่รับสินค้าเข้าสต็อก ระบบบันทึก:

- Supplier ที่ซื้อ
- วันที่รับสินค้า
- รายการสินค้า (link ไป product)
- จำนวน + Cost price ต่อชิ้น + Cost price รวม
- หลักฐาน (ใบเสร็จ/invoice จาก supplier)
- ผู้บันทึก

> เครื่องรุ่นเดียวกัน ซื้อจากคนละ supplier อาจได้ราคาทุนต่างกัน — ระบบเก็บ cost price ระดับ **individual item** (ไม่ใช่ระดับ model)

### 20.3 Purchase Order (PO)

**Flow:**

```
สร้าง PO (ระบุ supplier + รายการ + จำนวน + ราคา)
    ↓
DRAFT → เจ้าของ/ผู้จัดการ review
    ↓
APPROVED → ส่ง PO ให้ supplier (พิมพ์ PDF / ส่ง LINE)
    ↓
รับสินค้า (ทั้งหมด หรือ บางส่วน)
    ↓
PARTIALLY_RECEIVED / FULLY_RECEIVED
    ↓
สินค้าเข้าสต็อก → ตรวจเช็ค (ถ้ามือสอง) → พร้อมขาย
```

**สถานะ PO:** `DRAFT` → `APPROVED` → `PARTIALLY_RECEIVED` / `FULLY_RECEIVED` / `CANCELLED`

**ข้อมูล PO:** เลข PO (auto), Supplier, วันที่สั่ง, วันที่คาดรับ, รายการสินค้า (ยี่ห้อ/รุ่น/จำนวน/ราคา), ยอดรวม, หมายเหตุ, ผู้สร้าง/ผู้อนุมัติ

### 20.4 ราคาขายหลายราคา (Multi-Price)

สินค้าแต่ละชิ้นมี **หลายราคาขาย** กำหนดอิสระ:

```
Samsung Galaxy S24 Ultra (Grade A)
├── ราคาทุน: ฿10,000 (จาก Supplier A)
├── ราคาเงินสด: ฿12,900
├── ราคาผ่อน: ฿14,500  ★ default
├── ราคาพนักงาน: ฿11,500
└── ราคาส่ง: ฿11,000
```

**โครงสร้าง:**
- `cost_price` — ราคาทุน (จาก supplier/PO)
- `prices[]` — Array ของราคาขาย กำหนดได้ไม่จำกัด
  - `label` — ชื่อราคา (Admin กำหนดเอง)
  - `amount` — จำนวนเงิน
  - `is_default` — ราคาที่ใช้เป็นค่าเริ่มต้นตอนสร้างสัญญาผ่อน

**กฎ:**
- ต้องมีอย่างน้อย 1 ราคาขาย
- ราคา default ใช้ตอนสร้างสัญญา (พนักงานเปลี่ยนได้)
- Admin กำหนด price label ได้เอง (ไม่ fix ตายตัว)
- กำไรต่อชิ้น = ราคาขายที่ใช้ − ราคาทุน (แสดงในรายงาน)

## 23. Workflow เครื่องยึดคืน & Resell

### 19.1 กระบวนการ

```
ลูกค้า DEFAULT
    ↓
เจ้าของตัดสินใจยึดเครื่อง
    ↓
บันทึกการยึดคืนในระบบ
  - สาเหตุ
  - สภาพเครื่อง (grade: A/B/C/D)
  - ภาพถ่ายเครื่อง
    ↓
ตีราคาเครื่อง (appraisal)
  - ราคาตีโดยผู้จัดการ/เจ้าของ
  - เทียบกับยอดค้างชำระ → คำนวณกำไร/ขาดทุน
    ↓
สินค้าสถานะ → REPOSSESSED
    ↓
ซ่อม/ตรวจสภาพ (ถ้าจำเป็น)
  - บันทึกค่าซ่อม
  - สถานะ → REFURBISHED
    ↓
ตั้งราคาขายต่อ (resell price)
    ↓
วางขาย → IN_STOCK (เครื่องมือสอง)
    ↓
ขายได้ → SOLD_RESELL หรือ SOLD_INSTALLMENT (ถ้าขายผ่อนอีกรอบ)
```

### 19.2 ข้อมูลที่บันทึก

| ข้อมูล | รายละเอียด |
|---|---|
| วันที่ยึดคืน | timestamp |
| สัญญาต้นทาง | link ไป contract ที่ default |
| สภาพเครื่อง | Grade A (ดีมาก) / B (ดี) / C (พอใช้) / D (เสียหาย) |
| ราคาตี (appraisal) | จำนวนเงิน + ผู้ตี |
| ค่าซ่อม | จำนวนเงิน (ถ้ามี) |
| ราคาขายต่อ | จำนวนเงินที่ตั้งขาย |
| ภาพถ่าย | ภาพสภาพเครื่องตอนยึดคืน |

### 19.3 การคำนวณกำไร/ขาดทุน

```
ยอดค้างชำระ       = เงินต้นคงเหลือ + ดอกเบี้ยค้าง + ค่าปรับ
มูลค่ายึดคืน       = ราคาตี - ค่าซ่อม
ขาดทุนจากสัญญา    = ยอดค้างชำระ - มูลค่ายึดคืน
กำไรจาก resell    = ราคาขายต่อ - (ราคาตี + ค่าซ่อม)
```
