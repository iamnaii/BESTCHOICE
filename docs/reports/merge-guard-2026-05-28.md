#### вЪ†пЄП Warning вАФ `$executeRawUnsafe` used for advisory lock (diverges from security standards)

**File**: `apps/api/src/modules/contract-exchange/contract-exchange.service.ts` вАФ `nextExchangeContractNumber()`  
```typescript
const lockKey = this.hashLockKey(`exch:${yyyymmdd}`);
await tx.$executeRawUnsafe(`SELECT pg_advisory_xact_lock(${lockKey})`);
```
**Problem**: `$executeRawUnsafe` with string interpolation is flagged as a security anti-pattern (`.claude/rules/security.md` вАФ "SQL injection (unparameterized `$queryRaw`)"). In practice there is **no injection risk** here because `lockKey` is always a 32-bit integer produced by `hashLockKey` (which returns `number` via `| 0`). However, using `$executeRawUnsafe` instead of `Prisma.sql` tagged templates is inconsistent with how other advisory-lock usages in this codebase are expected to be written and will generate a false-positive if a security scanner runs on the diff.

**Recommended fix** (using `$executeRaw` with `Prisma.sql`):
```typescript
import { Prisma } from '@prisma/client';
// ...
await tx.$executeRaw(Prisma.sql`SELECT pg_advisory_xact_lock(${lockKey}::bigint)`);
```
Note: casting to `::bigint` is required because `pg_advisory_xact_lock` expects `bigint`, and Prisma parameterizes `number` as `int4` by default.
