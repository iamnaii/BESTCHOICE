### Warnings

**W1 — Duplicate CoA code mappings in `CATEGORY_CODE_MAP`**
```typescript
// accounting.service.ts
ADMIN_RENT: '53-1502',   // ค่าธรรมเนียมราชการ — TEMP placeholder
ADMIN_INSURANCE: '53-1502', // ค่าธรรมเนียมราชการ — TEMP placeholder
OTHER_MISC: '53-1502',   // ค่าธรรมเนียมราชการ
```
Three different expense categories map to the same CoA code `53-1502`. The description becomes the only differentiator in the ledger. This is intentional (noted as TEMP), but means the Trial Balance cannot distinguish rent/insurance/misc under this code. Should be resolved before A.7 adds dedicated accounts.

**W2 — `as any` casts on `expense.category`**
Three ESLint-disable `as any` casts in `accounting.service.ts` (create/update/findAll) pending Prisma client regeneration after migration. These are documented and bounded to Phase A.6. Should be removed once `npx prisma generate` is re-run post-migration.

**W3 — Category validation now accepts any non-empty string**
```typescript
// dto/expense.dto.ts
@IsString()
@IsNotEmpty()
category: string;
```
Was previously `@IsEnum(ExpenseCategory)`. Any non-empty string passes validation. Invalid CoA codes are caught at JE-post time (`resolveAccountCode` returns `null`), but no error is returned to the caller — the expense is created with a null `accountCode` and no JE is posted. Consider adding a server-side validation that the code exists in `chart_of_accounts`.

**W4 — `onModuleInit()` will crash startup if DB is empty / not yet seeded**
The boot validator queries `chart_of_accounts`. On a fresh environment running `prisma migrate deploy` without seeding first, this throws and blocks server startup. The wipe/reseed runbook covers this, but CI or fresh dev environments could hit this. Consider a graceful warning rather than hard throw, or document the deploy order in code.
